# Лекция 01: Архитектура ARM Cortex-M и семейство STM32

## 🎯 Цели урока

К концу этой лекции вы должны:

- Понимать, **что такое ARM** и почему практически все современные микроконтроллеры используют именно это ядро
- Отличать Cortex-M от других ARM-ядер и знать, **почему** для микроконтроллеров выбрали именно M-серию
- Представлять **общую структуру микроконтроллера**: из чего он состоит, как части взаимодействуют
- Уметь **читать маркировку** STM32 и осознанно выбирать чип под задачу
- Понимать **модель памяти** Cortex-M: почему адрес `0x08000000` — это Flash, а `0x40020000` — GPIO
- Знать, **что происходит** от нажатия кнопки Reset до выполнения первой строчки `main()`

## 1. Что такое микроконтроллер? Чем он отличается от микропроцессора?

Прежде чем мы начнём программировать, давайте разберёмся с фундаментальным вопросом: **что именно мы будем программировать?** Вы наверняка слышали слова «процессор», «микропроцессор», «микроконтроллер», «чип». Иногда их используют как синонимы, но это **не одно и то же**.

### 1.1. Микропроцессор — «мозг без рук»

**Микропроцессор** (Microprocessor, MPU — MicroProcessor Unit) — это центральное вычислительное устройство компьютера. Его задача — выполнять инструкции программы: складывать числа, сравнивать, пересылать данные, принимать решения.

**Классические примеры:** Intel Core i7, AMD Ryzen, Apple M1, ARM Cortex-A72 (в Raspberry Pi).

```
┌──────────────────────────────────┐
│       Микропроцессор             │
│                                  │
│  ┌──────────┐  ┌──────────┐     │
│  │   ALU    │  │  CU      │     │
│  │(Арифмет.-│  │(Control  │     │
│  │ логич.   │  │  Unit)   │     │
│  │  блок)   │  │          │     │
│  └──────────┘  └──────────┘     │
│          ┌──────────┐           │
│          │ Регистры │           │
│          └──────────┘           │
│                                  │
│  Больше внутри НЕГО ничего нет! │
└──────────────────────────────────┘
```

**Ключевая особенность:** микропроцессор — это **только** вычислительное ядро. У него нет собственной памяти, нет портов ввода-вывода, нет таймеров. Чтобы он заработал, нужно **внешнее окружение**:

| Что нужно микропроцессору         | Где находится     |
| --------------------------------- | ----------------- |
| Оперативная память (RAM)          | Отдельная микросхема на плате |
| Постоянная память (SSD/HDD)       | Отдельное устройство |
| Контроллеры ввода-вывода (USB, SATA, PCIe) | Чипсет на материнской плате |
| Видеокарта                        | Отдельная плата или интегрированная |
| Кварцевый генератор (тактирование) | Отдельный компонент |

**Аналогия:** Микропроцессор — это **профессор-математик**. Он блестяще решает уравнения, но не умеет готовить, убираться, ходить в магазин. Ему нужна команда помощников (чипсет, RAM, контроллеры), чтобы функционировать.

### 1.2. Микроконтроллер — «компьютер в одном чипе»

**Микроконтроллер** (Microcontroller, MCU — MicroController Unit) — это **полноценный компьютер, упакованный в один кристалл**. Внутри одного чипа находятся:

| Блок                   | Зачем нужен                                                  |
| ---------------------- | ------------------------------------------------------------ |
| **Процессорное ядро**  | Выполняет программу (так же, как микропроцессор)             |
| **Flash-память**       | Хранит программу (как SSD, только встроенный)                |
| **SRAM**               | Оперативная память для переменных (как RAM, только встроенная) |
| **GPIO**               | Цифровые входы/выходы — «руки» для взаимодействия с миром    |
| **Таймеры**            | Измерение времени, генерация ШИМ, подсчёт событий            |
| **Коммуникации**       | UART, SPI, I2C, USB, CAN — общение с другими устройствами    |
| **АЦП/ЦАП**            | Преобразование аналоговых сигналов в цифровые и обратно      |
| **DMA**                | Прямой доступ к памяти — пересылка данных без ядра           |
| **Система тактирования** | Генерация и распределение тактовых сигналов                |
| **Контроллер прерываний** | Обработка внешних и внутренних событий                    |

```
┌──────────────────────────────────────────────────────────┐
│                 Микроконтроллер STM32                     │
│                                                          │
│  ┌──────────┐  ┌──────────┐  ┌──────────────────────┐   │
│  │  Ядро    │  │  Flash   │  │       SRAM           │   │
│  │Cortex-M4 │─▶│ (програм-│  │  (данные, стек,      │   │
│  │          │  │   ма)    │  │    куча)             │   │
│  └────┬─────┘  └──────────┘  └──────────┬───────────┘   │
│       │                                 │               │
│       └───────────────┬─────────────────┘               │
│                       │                                 │
│  ┌────────────────────┴────────────────────────────┐    │
│  │                 Периферия                       │    │
│  │                                                 │    │
│  │  ┌──────┐ ┌──────┐ ┌──────┐ ┌──────┐ ┌──────┐  │    │
│  │  │ GPIO │ │ UART │ │ SPI  │ │ I2C  │ │ TIM  │  │    │
│  │  └──────┘ └──────┘ └──────┘ └──────┘ └──────┘  │    │
│  │  ┌──────┐ ┌──────┐ ┌──────┐ ┌──────┐ ┌──────┐  │    │
│  │  │ ADC  │ │ DAC  │ │ DMA  │ │ USB  │ │  RCC │  │    │
│  │  └──────┘ └──────┘ └──────┘ └──────┘ └──────┘  │    │
│  └─────────────────────────────────────────────────┘    │
│                                                          │
│  Всё это — в одном кристалле размером 10×10 мм          │
└──────────────────────────────────────────────────────────┘
```

**Аналогия:** Микроконтроллер — это **домашний мастер-универсал**. Он и считает, и провода подключает, и датчики читает, и моторы крутит. Всё в одном корпусе. Достал из коробки — и уже работает.

### 1.3. Сравнение: микропроцессор vs микроконтроллер

**Примеры встраиваемых микропроцессоров (MPU):**
- **NXP i.MX6ULL** (Cortex-A7, 528 МГц) — терминалы оплаты, HMI-панели
- **Allwinner H3** (Cortex-A7, 1.2 ГГц) — Orange Pi, TV-боксы
- **Broadcom BCM2837** (Cortex-A53, 1.2 ГГц) — Raspberry Pi 3
- **Rockchip RK3568** (Cortex-A55, 2 ГГц) — маршрутизаторы, шлюзы

**Примеры микроконтроллеров (MCU):**
- **STM32F407** (Cortex-M4, 168 МГц)
- **ESP32-S3** (Xtensa LX7, 240 МГц)
- **ATmega328P** (AVR, 20 МГц)

| Критерий                  | Микропроцессор (MPU)                   | Микроконтроллер (MCU)                      |
| ------------------------- | -------------------------------------- | ------------------------------------------ |
| **Что внутри чипа**       | Только ядро CPU (+ кэш)                | Ядро + Flash + SRAM + периферия (GPIO, TIM, UART, SPI, I2C, ADC…) |
| **Внешняя RAM**           | Обязательна (DDR3/DDR4, отдельная микросхема) | Не нужна (SRAM встроена)                   |
| **Внешняя Flash**         | Обязательна (eMMC, SD, SPI Flash)      | Не нужна (Flash встроена)                  |
| **Количество чипов на плате** | 5-10+ (CPU + DDR + Flash + PMIC + PHY) | 1-2 (MCU + пару конденсаторов)            |
| **Сложность PCB**         | 4-8 слоёв, DDR-трассировка, BGA-корпус | 2 слоя, LQFP/DIP корпус, справится новичок |
| **Частота**               | 500 МГц — 2+ ГГц                       | 8 МГц — 500 МГц                            |
| **Память**                | 128 МБ — 8 ГБ DDR + гигабайты eMMC     | 2 КБ — 2 МБ SRAM + 16 КБ — 2 МБ Flash      |
| **Операционная система**  | Linux, Android, Windows IoT            | Без ОС (bare-metal), FreeRTOS, Zephyr      |
| **Потребление**           | 0.5 — 10 Вт                            | 10 мкА — 100 мА (мкВт — мВт)              |
| **Время запуска**         | 5-30 секунд (загрузка ОС с SD/eMMC)    | 5-100 мс (сразу в main())                  |
| **Реакция на прерывание** | Микросекунды — миллисекунды (через ОС, кэш) | 12 тактов (напрямую, через NVIC)         |
| **Детерминизм**           | Нет (MMU, кэш, планировщик ОС вносят задержки) | Да (фиксированное время отклика)       |
| **Цена (чип + обвязка)**  | $10-50 (процессор + DDR + Flash + PMIC) | $0.20-20 (всё в одном чипе)               |
| **Типичное применение**   | Видеорегистраторы, терминалы, маршрутизаторы, планшеты | Датчики, пульты, стиральные машины, дроны, BMS |

**Простой пример для понимания:**

| Устройство | Что внутри | Почему |
|---|---|---|
| Видеорегистратор | NXP i.MX6 (MPU) + DDR3 + eMMC | Нужно сжимать видео (H.264), запускать Linux, писать на флешку |
| Датчик температуры в регистраторе | STM32L0 (MCU) | Просто читать датчик, отправить данные раз в секунду, спать остальное время |

> **Ключевое правило embedded:** если задача простая (читать датчик, мигнуть LED, отправить байт) — берёте **MCU**. Если задача сложная (видео, сетевой стек, графический интерфейс, файловая система) — берёте **MPU**. А часто — **оба**: MPU управляет системой, а MCU обслуживает датчики и исполнительные механизмы.

### 1.4. А что такое SoC?

**SoC** (System on Chip — «система на кристалле») — это промежуточный вариант. По сути, это микроконтроллер, но **гораздо более мощный**.

Типичный SoC содержит:
- Мощное процессорное ядро (Cortex-A, не Cortex-M)
- GPU (графический процессор)
- Контроллер внешней памяти (DDR)
- Видео-кодек, аудио-кодек
- Сетевые интерфейсы (Ethernet, WiFi, Bluetooth)
- И при этом — немного периферии для управления «железом» (GPIO, SPI, I2C)

**Примеры:**
- **Raspberry Pi (BCM2711)** — SoC с Cortex-A72, 4-8 ГБ RAM, GPU, USB, Ethernet. Запускает Linux.
- **ESP32** — SoC с Xtensa-ядром, WiFi, Bluetooth, GPIO, SPI, I2C. Работает без ОС или с FreeRTOS.
- **STM32MP1** — SoC от ST: Cortex-A7 (Linux) + Cortex-M4 (bare-metal) в одном чипе.

**Где граница?** Грубо говоря:
- **MCU** — нет MMU, нет Linux, килобайты памяти → микроконтроллер
- **SoC** — есть MMU, может Linux, мегабайты/гигабайты памяти → одноплатный компьютер
- **MPU** — только ядро, всё остальное внешнее → процессор

### 1.5. Где мы используем микроконтроллеры в повседневной жизни?

Вы удивитесь, но **в среднем доме 50-200 микроконтроллеров**. Вот лишь некоторые:

| Устройство                    | Что делает МК                                       |
| ----------------------------- | --------------------------------------------------- |
| Стиральная машина             | Управление мотором, чтение датчиков, отображение    |
| Пульт от телевизора           | Сканирование кнопок, отправка ИК-сигнала            |
| Электронные весы              | Чтение тензодатчика (ADC), отображение на LCD       |
| Беспроводные наушники         | Приём Bluetooth, обработка аудио (DSP), зарядка     |
| Автомобиль (современный)      | 50-100 МК: ABS, подушки безопасности, климат-контроль |
| Климат-контроль               | Чтение температуры, управление вентилятором, дисплей |
| Дрон                          | Стабилизация (гироскоп), управление моторами (ШИМ)  |
| Умная лампочка                | WiFi, ШИМ для RGB, протокол управления              |
| Банковская карта              | Один из самых маленьких МК — Secure Element         |

**Общий паттерн:** микроконтроллер **читает датчики → принимает решение → управляет исполнительными устройствами**. Именно этим мы и будем заниматься в этом курсе.

### 1.6. Архитектуры процессоров — какие бывают?

Микроконтроллеры строятся на разных **архитектурах** (системах команд). Архитектура определяет:
- Какие инструкции понимает процессор
- Как организованы регистры и память
- Как работает стек и вызов функций

Вот основные архитектуры, которые вы встретите в мире микроконтроллеров:

#### ARM (Advanced RISC Machine)

**Самая популярная архитектура** в мире микроконтроллеров. 90% всех современных 32-битных МК используют ARM.

| Характеристика | Значение |
|---|---|
| Тип | RISC (Reduced Instruction Set) |
| Разрядность | 32-бит (Cortex-M), 64-бит (Cortex-A AArch64) |
| Модель памяти | Гарвардская модифицированная (раздельные шины для инструкций и данных) |
| Набор команд | Thumb (16-бит), Thumb-2 (16/32-бит), ARM (32-бит, не в Cortex-M) |
| Лицензирование | ARM продаёт лицензии ядра, производители добавляют периферию |
| Примеры МК | STM32, NXP LPC, Nordic nRF, SAMD, TM4C |

**Почему ARM доминирует:**
- Единая экосистема: один компилятор (GCC, LLVM) работает на всех ARM-чипах
- Отличная документация и сообщество
- Низкое энергопотребление
- Масштабируемость: от M0+ (ультра-дешёвый) до M7 (высокая производительность)

#### AVR

8-битная архитектура от Atmel (сейчас Microchip). Знаменитая платформа Arduino Uno построена на AVR (ATmega328P).

| Характеристика | Значение |
|---|---|
| Тип | RISC |
| Разрядность | 8-бит |
| Память | Гарвардская (отдельное адресное пространство для Flash и SRAM) |
| Примеры | ATmega328P, ATmega2560, ATtiny85 |

**Плюсы:** Простой, дешёвый, огромное сообщество (Arduino).
**Минусы:** 8-бит = сложно работать с большими числами, мало памяти, нет FPU.

#### PIC

8/16/32-битная архитектура от Microchip. Одна из старейших в мире микроконтроллеров.

| Характеристика | Значение |
|---|---|
| Тип | Гарвардская архитектура (раздельные шины инструкций и данных) |
| Разрядность | 8/16/32 бит |
| Примеры | PIC16F877A, PIC18F4550, PIC32 |

**Особенность:** Гарвардская архитектура означает, что инструкции и данные — на разных шинах. Это быстрее, но сложнее программировать.

#### RISC-V

Открытая (open-source) архитектура. Это **новая** альтернатива ARM, которая набирает обороты.

| Характеристика | Значение |
|---|---|
| Тип | RISC |
| Разрядность | 32-бит (RV32), 64-бит (RV64) |
| Лицензирование | Полностью открытая (бесплатная), любой может реализовать ядро |
| Примеры | GD32V, ESP32-C3, HiFive1 |

**Почему это важно:** ARM — проприетарная (закрытая) архитектура. RISC-V — открытая. Если когда-либо ARM поднимет цены на лицензии, RISC-V станет массовой альтернативой.

#### x86 / x64

Архитектура Intel/AMD. **Не используется в микроконтроллерах** (слишком мощная, слишком прожорливая). Но вы должны знать о ней для контраста.

| Характеристика | Значение |
|---|---|
| Тип | CISC (Complex Instruction Set) |
| Разрядность | 32-бит (x86), 64-бит (x64) |
| Примеры | Intel Core i7, AMD Ryzen |
| Где используется | ПК, серверы, ноутбуки |

### 1.7. RISC vs CISC — коротко и по делу

Вы уже встречали эти аббревиатуры. Давайте разберёмся без академической мути.

**CISC** (Complex Instruction Set Computer) — процессор с богатым набором сложных команд. Пример: x86.

- Команды делают много за один раз: «загрузи строку из памяти, преобразуй в число, сложи, запиши обратно» — всё одной инструкцией
- Команды разной длины (1-15 байт)
- Команда может выполняться 10-20 тактов
- Сложнее проектировать конвейер (pipeline)

**RISC** (Reduced Instruction Set Computer) — процессор с минимальным набором простых команд. Пример: ARM.

- Команды простые: «загрузи», «сложи», «запиши». Каждая — один такт (в идеале)
- Все команды одной длины (16 или 32 бита)
- Конвейер работает отлично: пока одна инструкция выполняется, следующая уже декодируется
- Проще ядро → меньше транзисторов → дешевле чип, меньше энергопотребление

```c
// CISC (x86) — одна инструкция, но 8-15 тактов
ADD DWORD PTR [memory], 42

// RISC (ARM) — три простые инструкции, но каждая 1 такт (итого 3 такта + конвейер)
LDR  r0, [memory]    // загрузить из памяти в регистр r0
ADD  r0, r0, #42     // сложить 42, результат в r0
STR  r0, [memory]    // записать r0 обратно в память
```

Для микроконтроллера, где важна **предсказуемость времени выполнения** и **энергоэффективность**, RISC подходит идеально.

## 2. ARM — что это вообще такое и почему все его используют?

### 2.1. ARM не производит чипы

Давайте начнём с самого удивительного факта для новичков. Компания **ARM Holdings** (сейчас принадлежит SoftBank) — **не имеет ни одного завода**. Она не паяет чипы, не продаёт отладочные платы. Она продаёт **интеллектуальную собственность** — чертежи процессорных ядер.

Представьте, что ARM — это архитектурное бюро. Оно рисует проект дома, но не строит его. А строители — это уже STMicroelectronics, NXP, Microchip, Nordic и ещё десятки компаний.

```
ARM Holdings (проект ядра)
    │
    ├── лицензия ──→ STMicroelectronics ──→ STM32
    ├── лицензия ──→ NXP                  ──→ LPC, Kinetis
    ├── лицензия ──→ Microchip            ──→ SAM
    ├── лицензия ──→ Nordic               ──→ nRF
    └── лицензия ──→ Texas Instruments    ──→ Tiva-C
```

**Почему это важно для вас как разработчика?** Потому что, научившись программировать STM32, вы автоматически понимаете 80% любого другого ARM-микроконтроллера. Ядро одинаковое. Различается только «обвес» — периферия, которую добавляет производитель.

### 2.2. Зачем мир перешёл на ARM?

Ещё в 2000-х годах доминировали архитектуры 8051 (Intel), AVR (Atmel), PIC (Microchip). Почему ARM победил?

|Причина|Что это значит на практике|
|---|---|
|**RISC-архитектура**|Меньше команд, но каждая выполняется быстрее. AVR — 130+ команд, ARM — ~20 базовых, но конвейеризированных|
|**32-битность**|Работа с 32-битными числами за один такт. AVR/8051 — 8 бит, 32-битное сложение = 4 операции|
|**Единая экосистема**|Один компилятор (GCC, LLVM), один отладчик (GDB, OpenOCD), один формат отладки (SWD/JTAG)|
|**Низкая цена ядра**|Лицензия стоит денег, но в пересчёте на миллионные тиражи — центы за чип|
|**Энергоэффективность**|ARM проектировался для мобильных устройств с самого начала|

> **Аналогия:** Представьте, что AVR — это механическая коробка передач. Много ручных операций, полный контроль, но нужно переключать каждую передачу. ARM — это автомат: меньше команд, но каждая работает оптимально. В городе (встраиваемые задачи) автомат часто лучше.

### 2.3. RISC vs CISC — коротко и по делу (повторение)

> Этот подраздел дублирует раздел 1.7 для связности. Если вы уже прочитали про RISC/CISC выше — можете пропустить.

Вы наверняка слышали эти аббревиатуры. Давайте разберёмся без академической мути.

**CISC** (Complex Instruction Set Computer) — процессор с богатым набором команд. Пример: x86 (ваш настольный ПК). У него есть команды вроде «загрузи строку из памяти, преобразуй в число, сложи, запиши обратно» — всё одной инструкцией. Звучит круто? Да, но:

- Команда может выполняться 10-20 тактов
- Конвейер (pipeline) работает плохо, потому что команды разной длины
- Сложнее проектировать

**RISC** (Reduced Instruction Set Computer) — процессор с минимальным набором команд. Пример: ARM. У него команды простые: «загрузи», «сложи», «запиши». Каждая — один такт. Почему это лучше для микроконтроллеров?

- **Предсказуемость:** каждая инструкция = 1 такт (в идеале). Легче рассчитывать real-time.
- **Конвейер:** пока одна инструкция выполняется, следующая уже декодируется. Как на заводе — детали движут по ленте.
- **Проще ядро:** меньше транзисторов → дешевле чип, меньше энергопотребление.

```c
// CISC (x86) — одна инструкция, но 8-15 тактов
ADD DWORD PTR [memory], 42

// RISC (ARM) — три инструкции, но каждая 1 такт (итого 3 такта + конвейер)
LDR  r0, [memory]    // загрузить из памяти в регистр
ADD  r0, r0, #42     // сложить
STR  r0, [memory]    // записать обратно
```

Для микроконтроллера, где важна **предсказуемость времени выполнения** и **энергоэффективность**, RISC подходит идеально.

## 3. Семейство Cortex: M, A, R — в чём разница?

ARM делит свои ядра на три большие группы. Понимание этой классификации — ключ к осознанному выбору чипа.

### 3.1. Cortex-A — Application processors

Это «большие» процессоры. Cortex-A53, A72, A78 — они стоят в вашем телефоне, планшете, Raspberry Pi.

|Характеристика|Значение|
|---|---|
|Разрядность|64-бит (AArch64)|
|ОС|Linux, Android, Windows|
|Частота|1-3+ ГГц|
|Память|Внешняя DDR (гигабайты)|
|MMU|✅ Полная (виртуальная память)|
|Запуск|Секунды (загрузка ОС)|

**Ключевая особенность:** у них есть **MMU** (Memory Management Unit). Это блок, который обеспечивает виртуальную память — каждую программу в своём адресном пространстве, защита от падения одного процесса на другой. Без MMU Linux не работает.

**Минус для микроконтроллеров:** MMU добавляет задержки. Прерывание обрабатывается не мгновенно, а через переключение контекста. Для управления мотором или обработки датчика это неприемлемо.

### 3.2. Cortex-R — Real-time processors

Специализированные процессоры для жёсткого реального времени. Стоят в жёстких дисках, автомобильных тормозных системах, авионике.

|Характеристика|Значение|
|---|---|
|Детерминизм|Прерывание обрабатывается за фиксированное число тактов|
|Надёжность|Дублирование логики, ECC-память|
|Частота|400-600 МГц|
|ОС|FreeRTOS, INTEGRITY, или bare-metal|

**Главное:** гарантированное время отклика. Если в вашем устройстве пропуск обработки прерывания более чем на 500 нс приводит к катастрофе — вам нужен Cortex-R.

### 3.3. Cortex-M — Microcontrollers

И вот мы добрались до нашей темы. Cortex-M — это ядра для микроконтроллеров. Они занимают золотую нишу между A и R:

|Характеристика|Значение|
|---|---|
|Разрядность|32-бит|
|Частота|48 МГц (M0+) — 480 МГц (H7)|
|Память|Встроенная Flash (16 КБ — 2 МБ), SRAM (4 КБ — 1 МБ)|
|MMU|❌ Нет (в большинстве)|
|Запуск|Миллисекунды|
|Прерывания|NVIC — до 240 прерываний, приоритеты|
|ОС|Bare-metal, FreeRTOS, Zephyr, RT-Thread|

**Почему Cortex-M для микроконтроллеров:**

1. **Детерминизм:** прерывание обрабатывается за 12 тактов (M3/M4). Всегда. Вы точно знаете, когда код отреагирует на событие.
2. **Простота:** нет MMU, нет кэша (в большинстве), нет виртуальной памяти. Запустил — и через 50 мкс уже в `main()`.
3. **Энергоэффективность:** M0+ потребляет микроамперы в active режиме. M4 с FPU — миллиамперы при полной загрузке.
4. **Цена:** чипы от $0.5 (STM32F030) до $15 (STM32H743).

### 3.4. Линейка Cortex-M: от простого к сложному

Теперь разберём каждое ядро подробно. Это критически важно, потому что от выбора ядра зависит всё: набор инструкций, доступная периферия, сложность программирования.

#### Cortex-M0 / M0+ — ультра-бюджетные

Самое простое ядро. Создано, чтобы заменить 8-битные микроконтроллеры (AVR, 8051) с минимальной ценой.

**Что умеет:**
- Только 16-бит инструкции Thumb (полный Thumb-2 не поддерживает)
- Нет аппаратного деления (деление = программная библиотека, медленно)
- Нет битовой адресации (работа с отдельными битами через маски)
- 0.5-1 DMIPS/МГц (Dhrystone Millions of Instructions Per Second)

**Что это значит на практике:** Если вы пишете `a / b` в коде — компилятор вставит вызов библиотеки, которая будет делать деление циклом вычитания. Это сотни тактов.

**Типичные чипы:** STM32F030, STM32L011, STM32G031
**Когда использовать:** Простая логика, реле, датчик температуры, LED-контроллер. Всё, где не нужны вычисления.

#### Cortex-M3 — «рабочая лошадка»

Первое полноценное ARM-ядро для микроконтроллеров. Появилось в 2004 году и до сих пор используется.

**Что добавилось по сравнению с M0+:**
- Полный **Thumb-2** — смесь 16-бит и 32-бит инструкций. Код компактнее, производительность выше.
- **Аппаратное деление** — `SDIV`, `UDIV`. Деление за 2-12 тактов вместо сотен.
- **Битовая адресация** (Bit-Banding) — можно обращаться к отдельному биту как к 32-битному слову.
- **Аппаратный умножитель** — `MUL` за 1 такт.
- **NVIC с 240 прерываниями** и 256 уровнями приоритетов.
- Вложенные прерывания (прерывание может прервать другое прерывание).
- 1.25 DMIPS/МГц

**Типичные чипы:** STM32F103 (знаменитый «Blue Pill»), STM32L151, STM32F303
**Когда использовать:** Промышленная автоматика, контроллеры, устройства с несколькими протоколами связи.

#### Cortex-M4 — M3 + DSP + FPU

Это M3, к которому добавили два важных блока:

**1. DSP-расширения (Digital Signal Processing):**

- **SIMD** (Single Instruction, Multiple Data) — одна инструкция обрабатывает несколько данных одновременно.

  ```c
  // Обычное сложение (M3) — 4 такта
  c[0] = a[0] + b[0];
  c[1] = a[1] + b[1];
  c[2] = a[2] + b[2];
  c[3] = a[3] + b[3];

  // SIMD (M4) — 1 такт (QADD8)
  // Сложить 4 пары 8-битных чисел одной инструкцией
  ```

- **MAC** (Multiply-Accumulate) — аппаратное умножение с накоплением. Ключевая операция для FFT, FIR-фильтров, PID-регуляторов.

  ```c
  // FIR-фильтр: y[n] = Σ b[k] * x[n-k]
  // M3: каждая итерация = умножить + сложить = много тактов
  // M4: инструкция SMLAD (Signed Multiply-Long and Add Dual) = 1 такт
  ```

**2. FPU (Floating Point Unit) — аппаратные вычисления с плавающей точкой:**

Это самый важный момент для понимания. На M3 без FPU операция `float a = 3.14f * 2.71f;` превращается в **сотни тактов** программной эмуляции. На M4 с FPU — это **3 такта**.

> **Важно:** FPU — опциональный блок. STM32F407 **имеет** FPU (обозначается как FPUv4-SP). Но некоторые чипы на базе M4 могут его не иметь. Всегда проверяйте datasheet.

**Производительность:**
- С FPU: операции с `float` — 3-5 тактов
- Без FPU (M3): операции с `float` — 50-200 тактов
- Разница — **10-40x** для математического кода

**Типичные чипы:** STM32F407, STM32F411, STM32L476, STM32G474
**Когда использовать:** Всё, что связано с обработкой сигналов: аудио, мотор-контроль, фильтры, FFT, PID, сенсорные экраны.

#### Cortex-M7 — высокая производительность

Это вершина линейки M (до появления M55/M85 с AI-расширениями). M7 — это M4 на стероидах:

- **6-ступенчатый конвейер** (у M4 — 3 ступени). Больше инструкций параллельно.
- **Кэш инструкций (I-Cache) и данных (D-Cache)** — 16-32 КБ каждый. Критически важно для производительности!
- **Двойная точность FPU** (DP-FPU) — работает с `double` аппаратно, а не только с `float`.
- **До 2.14 DMIPS/МГц** — почти в 2 раза быстрее M4 на той же частоте.

**НО:** Кэш создаёт проблемы. Если вы записали данные в память, а потом периферия (например, DMA) их читает — может возникнуть рассинхронизация. Нужно управлять когерентностью кэша.

```c
// Перед отправкой данных через DMA — очистить кэш
SCB_CleanDCache_by_Addr((uint32_t*)buffer, size);
DMA_Start(buffer, size);

// После приёма данных через DMA — инвалидировать кэш
DMA_Receive(buffer, size);
SCB_InvalidateDCache_by_Addr((uint32_t*)buffer, size);
// Теперь buffer содержит актуальные данные
```

> **Предупреждение:** Это одна из самых частых ошибок при работе с M7. Данные в кэше и в RAM различаются, программист читает «старые» данные из кэша и часами ищет баг.

**Типичные чипы:** STM32H743, STM32H750
**Когда использовать:** Видеобработка, графические интерфейсы, машинное обучение, сетевые стеки (Ethernet, USB HS).

#### Cortex-M33/M35P — безопасность (TrustZone)

Добавлена поддержка **ARM TrustZone** — аппаратное разделение на «безопасный» и «небезопасный» мир. Критически важно для IoT-устройств, финансовых терминалов, медицинских приборов.

#### Сводная таблица

|Ядро|Thumb-2|Деление|DSP|FPU|Кэш|DMIPS/МГц|Типичная частота|
|---|---|---|---|---|---|---|---|
|M0+|❌ (16-bit только)|❌|❌|❌|❌|0.5-1|48 МГц|
|M3|✅|✅|❌|❌|❌|1.25|72-200 МГц|
|M4|✅|✅|✅|✅ (опц.)|❌|1.25|84-180 МГц|
|M7|✅|✅|✅|✅ (DP)|✅|2.14|200-480 МГц|
|M33|✅|✅|❌|✅ (опц.)|❌|1.0-1.5|64-100 МГц|

> **💡 Рекомендация для этого курса:** Мы работаем с **STM32F407** (Cortex-M4 с FPU). Это оптимальный выбор для обучения: достаточно мощный для серьёзных проектов, достаточно простой для понимания основ. Код будет совместим с F1 (M3), L4 (M4 low-power), G4 (M4 с усиленной аналоговой периферией).

## 4. Из чего состоит микроконтроллер?

Прежде чем погружаться в программирование, давайте поймём общую структуру. Что вообще находится внутри STM32?

### 4.1. Общая схема

Микроконтроллер — это **компьютер в одном чипе**. В отличие от вашего настольного ПК (где процессор, оперативная память, жёсткий диск, видеокарта — отдельные микросхемы), в микроконтроллере всё интегрировано:

```
┌──────────────────────────────────────────────────────────┐
│                   Микроконтроллер STM32                   │
│                                                          │
│  ┌──────────┐   ┌──────────┐   ┌──────────────────────┐  │
│  │   Ядро   │   │  Flash   │   │       SRAM           │  │
│  │ Cortex-M │──▶│ (програм-│   │   (данные, стек,     │  │
│  │          │   │   ма)    │   │     куча)            │  │
│  └────┬─────┘   └──────────┘   └──────────┬───────────┘  │
│       │                                   │              │
│       └───────────────┬───────────────────┘              │
│                       │ Шины (AHB, APB1, APB2)           │
│                       │                                  │
│  ┌────────────────────┴───────────────────────────────┐  │
│  │               Периферия                            │  │
│  │                                                    │  │
│  │  ┌──────┐ ┌──────┐ ┌──────┐ ┌──────┐ ┌─────────┐  │  │
│  │  │ GPIO │ │ UART │ │  SPI │ │  I2C │ │  Таймеры│  │  │
│  │  └──────┘ └──────┘ └──────┘ └──────┘ └─────────┘  │  │
│  │  ┌──────┐ ┌──────┐ ┌──────┐ ┌──────┐ ┌─────────┐  │  │
│  │  │  ADC │ │  DAC │ │  DMA │ │  USB │ │    CAN  │  │  │
│  │  └──────┘ └──────┘ └──────┘ └──────┘ └─────────┘  │  │
│  └────────────────────────────────────────────────────┘  │
│                                                          │
│  ┌────────────────────────────────────────────────────┐  │
│  │           Система тактирования (RCC)               │  │
│  │  (генераторы, PLL, делители — «сердцебиение» чипа) │  │
│  └────────────────────────────────────────────────────┘  │
│                                                          │
│  ┌────────────────────────────────────────────────────┐  │
│  │     Контроллер прерываний (NVIC)                   │  │
│  │  (диспетчер: какое прерывание обслужить первым)    │  │
│  └────────────────────────────────────────────────────┘  │
└──────────────────────────────────────────────────────────┘
```

Давайте разберём каждый блок.

### 4.2. Ядро (Core) — «мозг» системы

**Ядро** — это то, что выполняет вашу программу. Это процессор ARM Cortex-M. Он:

- Читает инструкции из Flash
- Выполняет арифметические и логические операции
- Работает с регистрами (быстрая память внутри ядра)
- Обрабатывает прерывания
- Управляет всей периферией (через запись/чтение регистров)

**Но!** Ядро — это только часть микроконтроллера. Без периферии, памяти и системы тактирования оно бесполезно.

### 4.3. Flash — «жёсткий диск» микроконтроллера

**Flash-паять** — это энергонезависимая память. Она хранит вашу программу, даже когда питание отключено. Когда вы «прошиваете» микроконтроллер — вы записываете данные именно сюда.

**Характеристики:**
- Размер: от 16 КБ (STM32F030) до 2 МБ (STM32H7)
- Скорость: медленнее, чем SRAM. Требует «wait states» (тактов ожидания)
- Запись: медленная, нужно стирать секторами. Поэтому нельзя просто так записать данные во Flash во время выполнения

> **Аналогия:** Flash — это SSD вашего микроконтроллера. Здесь живёт программа и константы.

### 4.4. SRAM — «оперативная память»

**SRAM** (Static Random Access Memory) — это оперативная память. Здесь живут все переменные во время выполнения: локальные переменные функций, глобальные переменные, стек, куча.

**Характеристики:**
- Быстрее, чем Flash
- Энергозависимая (при отключении питания данные теряются)
- Размер: от 4 КБ до 1 МБ (в зависимости от чипа)

> **Аналогия:** SRAM — это RAM вашего ПК. Здесь всё происходит «вживую».

### 4.5. Периферия — «органы чувств и действия»

**Периферия** — это всё, что позволяет микроконтроллеру взаимодействовать с внешним миром:

|Блок|Назначение|Пример|
|---|---|---|
|**GPIO** (General Purpose Input/Output)|Цифровые входы/выходы|Управление LED, чтение кнопок|
|**UART/USART**|Последовательный порт|Обмен данными с ПК, модулями Bluetooth|
|**SPI** (Serial Peripheral Interface)|Высокоскоростная шина|Дисплеи, SD-карты, внешняя Flash|
|**I2C** (Inter-Integrated Circuit)|Двухпроводная шина|Датчики температуры, давления, EEPROM|
|**ADC** (Analog-to-Digital Converter)|Аналого-цифровой преобразователь|Чтение аналоговых датчиков|
|**DAC** (Digital-to-Analog Converter)|Цифро-аналоговый преобразователь|Генерация аналоговых сигналов|
|**Таймеры (TIM)**|Счётчики, ШИМ, захват сигнала|Управление моторами, измерение импульсов|
|**DMA** (Direct Memory Access)|Прямой доступ к памяти|Копирование данных без участия ядра|
|**USB**|Universal Serial Bus|Подключение к ПК как USB-устройство|
|**CAN**|Controller Area Network|Автомобильная промышленность|

> **Аналогия:** Если ядро — это мозг, то периферия — это глаза, уши, руки и ноги. Без периферии микроконтроллер — просто калькулятор, замкнутый на себя.

### 4.6. Система тактирования (RCC) — «сердцебиение» чипа

**RCC** (Reset and Clock Control) — это блок, который генерирует и распределяет тактовые сигналы по всему чипу.

Откуда берётся тактовая частота?

```
┌─────────────────────────────────────────────────┐
│              Система тактирования                │
│                                                 │
│  HSI (16 МГц) ──┐                               │
│  (внутренний    │                                │
│   RC-генератор)  │                               │
│                  ▼                               │
│  HSE (8 МГц) ──►│  ┌──────┐                     │
│  (внешний       ├──►│ PLL  │──► SYSCLK (168 МГц)│
│   кварц)        │  │      │                     │
│                 │  └──────┘                     │
│                 │     │                          │
│                 │     ├──► AHB (168 МГц)         │
│                 │     ├──► APB1 (42 МГц)         │
│                 │     └──► APB2 (84 МГц)         │
└─────────────────┴────────────────────────────────┘
```

**Почему несколько шин?** Потому что не всей периферии нужна высокая частота. Высокоскоростная (ADC, SPI1) — на быстрой шине (APB2). Низкоскоростная (I2C, USART) — на медленной (APB1). Это экономит энергию и упрощает дизайн.

> **Критически важно:** Без правильной настройки RCC ваш микроконтроллер работает на внутреннем генераторе (HSI, 16 МГц) вместо 168 МГц. Это **в 10 раз медленнее**! Один из первых вопросов при отладке: «А вы уверены, что тактирование настроено?»

### 4.7. Контроллер прерываний (NVIC) — «диспетчер»

**NVIC** (Nested Vectored Interrupt Controller) — это «секретарь», который решает, какое прерывание обработить первым.

Представьте: вы работаете (main()), вдруг:
- Телефон звонит (USART прерывание — пришли данные)
- Дверь звонит (EXTI прерывание — кнопка нажата)
- Таймер сработал (SysTick — прошла 1 мс)

NVIC решает: что важнее, в каком порядке обрабатывать, можно ли одно прерывание прервать другим.

## 5. Модель памяти Cortex-M

Один из самых важных разделов. Понимание карты памяти — ключ к пониманию того, как работает микроконтроллер.

### 5.1. Единое 32-битное адресное пространство

Cortex-M имеет **4 ГБ** адресного пространства (2³² = 4 294 967 296 байт). Оно разделено на 8 блоков по 512 МБ:

```
Адрес               Размер    Содержание                    Доступ
─────────────────────────────────────────────────────────────────
0x0000 0000         512 МБ    Code (Flash, boot)            eXecute
0x2000 0000         512 МБ    SRAM                          Read/Write
0x4000 0000         512 МБ    Peripherals (APB/AHB)         Read/Write
0x6000 0000         512 МБ    External memory (FSMC)        Read/Write
0x8000 0000         512 МБ    External memory (FSMC)        Read/Write
0xA000 0000         512 МБ    External memory (FSMC)        Read/Write
0xC000 0000         512 МБ    External memory (FSMC)        Read/Write
0xE000 0000         512 МБ    Internal peripherals (NVIC)   Read/Write
0xF000 0000         256 МБ    Reserved (ARM)                —
0xF800 0000         240 МБ    Reserved (Vendor)             —
```

**Почему это важно:** В отличие от 8-битных микроконтроллеров (где у Flash одни адреса, у RAM другие, у периферии третьи — и доступ к ним через специальные инструкции), Cortex-M использует **memory-mapped I/O**. Всё — просто адреса в памяти.

```c
// На AVR (гарвардская архитектура):
uint8_t x = pgm_read_byte(&flash_data);  // Специальная инструкция для Flash
PORTB = value;                           // Специальная инструкция для I/O

// На Cortex-M (фон-Нейман):
uint8_t x = flash_data;                  // Просто чтение из памяти
GPIOA->ODR = value;                      // Просто запись по адресу
```

### 5.2. Детальная карта памяти STM32F407

Давайте посмотрим, как конкретно размещены компоненты в STM32F407:

```
0xE00F FFFF ────────────────────────────────┐
              Core Peripherals (NVIC,        │
              SysTick, SCB, ...)             │  System bus
0xE000 0000 ────────────────────────────────┤
                                            │
0x9FFF FFFF ────────────────────────────────┤
              External memory                │
0x6000 0000 ────────────────────────────────┘
              (FSMC — Flexible Static
0x5FFF FFFF ────────────────────────────────┐ Memory Controller)
                                            │
0x4002 4C00                                 │  AHB1
0x4002 4BFF ────────────────────────────────┤  Peripherals
              GPIOA-GPIOI                   │  (DMA, Flash,
0x4002 0000 ────────────────────────────────┤  RCC, ...)
                                            │
0x4001 4C00                                 │  APB2
0x4001 4BFF ────────────────────────────────┤  Peripherals
              USART1, USART6,               │  (высокоскоростные)
0x4001 0000 ────────────────────────────────┤
              SPI1, ADC1-3, TIM1, TIM8      │
0x4000 4C00                                 │  APB1
0x4000 4BFF ────────────────────────────────┤  Peripherals
              USART2-3, SPI2-3,             │  (низкоскоростные)
0x4000 0000 ────────────────────────────────┤  TIM2-7, I2C1-3
                                            │
0x3000 0000                                 │  Bit-Banding alias
0x2200 0000 ────────────────────────────────┤
                                            │
0x2002 0000                                 │  Core Coupled Memory
0x2001 FFFF ────────────────────────────────┤  (64 КБ, быстрый доступ)
                                            │
0x2001 0000                                 │  General SRAM
0x2000 FFFF ────────────────────────────────┤  (112 КБ)
0x2000 0000 ────────────────────────────────┤
                                            │
0x1FFF FFFF                                 │  System Memory (bootloader)
0x1FFF 0000 ────────────────────────────────┤  (ST factory bootloader)
                                            │
0x080F FFFF                                 │  Flash (1 МБ)
0x0800 0000 ────────────────────────────────┤  (здесь живёт ваша программа)
                                            │
0x000F FFFF                                 │  Alias (отображение Flash/SRAM)
0x0000 0000 ────────────────────────────────┘  (зависит от BOOT0/BOOT1)
```

Давайте разберём каждую область.

#### Flash (0x0800 0000)

Здесь хранится ваша программа. Когда вы «прошиваете» микроконтроллер — вы записываете данные именно по этому адресу.

**Особенности Flash:**
- **Read-only** во время выполнения (нельзя выполнить `*(uint32_t*)0x08000000 = 42`)
- **eXecute-In-Place (XIP)** — процессор выполняет код прямо из Flash, без копирования в RAM
- **Wait states:** Flash медленнее процессора. При 168 МГц нужно 5 wait states (6 тактов на чтение). Кэш-предвыборка помогает, но не всегда.
- **Разделение:** 1 МБ Flash разбит на секторы (4x16 КБ + 1x64 КБ + 7x128 КБ). Это важно для OTA-обновлений.

```c
// Программа находится в Flash по адресу 0x08000000+
// Все функции, константы, строки — здесь

const char message[] = "Hello";  // Во Flash (если const)
char buffer[] = "Hello";         // В SRAM (копируется из Flash при старте)

// Нельзя записать во Flash напрямую!
// *(uint32_t*)0x08001000 = 42;  // BusFault!
// Для записи нужно: unlock → erase → program → lock
```

#### SRAM (0x2000 0000)

Оперативная память. Здесь живут все переменные во время выполнения.

STM32F407 имеет **192 КБ SRAM**, разделённой на две области:

|Область|Адрес|Размер|Особенности|
|---|---|---|---|
|General SRAM|0x2000 0000|112 КБ|Основная RAM для переменных|
|CCM (Core Coupled Memory)|0x1000 0000|64 КБ|Быстрая RAM, доступна только ядру|

**CCM RAM** — специальная быстрая память, подключённая напрямую к ядру (через D-Code bus). DMA не имеет к ней доступа! Это важно.

```c
// В linker-скрипте (.ld):
/* Обычная SRAM — доступна и ядру, и DMA */
.ram (RW) : ORIGIN = 0x20000000, LENGTH = 112K

/* CCM RAM — только ядро, DMA НЕ работает! */
.ccmram (RW) : ORIGIN = 0x10000000, LENGTH = 64K

// Переменная в CCM (GCC):
__attribute__((section(".ccmram")))
uint16_t adc_buffer[1024];  // Быстрая RAM, но DMA не скопирует
```

> **Практическое правило:** Кладите в CCM RAM: стек, локальные переменные ISR, быстрые буферы. В обычную SRAM: буферы для DMA, глобальные данные, кучу (heap).

#### Peripherals (0x4000 0000 — 0x4002 4BFF)

Здесь живут регистры периферии. Каждый регистр — 32-битное слово по конкретному адресу.

**Две шины периферии:**

|Шина|Скорость|Периферия|
|---|---|---|
|**APB1** (до 42 МГц)|Низкая|USART2-3, SPI2-3, I2C1-3, TIM2-7, WWDG, PWR|
|**APB2** (до 84 МГц)|Высокая|USART1, USART6, SPI1, ADC1-3, TIM1, TIM8, SYSCFG, все GPIO|
|**AHB1** (до 168 МГц)|Самая быстрая|DMA1-2, RCC, Flash, CRC, все GPIO (порты A-I)|

**Почему разделение?** Высокоскоростная периферия (ADC, SPI1, TIM1) подключена к быстрой шине. Низкоскоростная (I2C, USART) — к медленной. Это оптимизирует общую пропускную способность.

#### System Memory (0x1FFF 0000)

Здесь «зашит» заводской **bootloader** ST. Он умеет:
- Загружать прошивку через USART1
- Загружать через USB (DFU)
- Загружать через CAN

Используется, когда BOOT0=1, BOOT1=0. Удобно, если вы «забриковали» чип и не можете прошить через SWD.

### 5.3. Memory-Mapped I/O — как работают регистры периферии

Ключевая концепция: регистры периферии — это просто **адреса в памяти**. Запись по адресу = команда периферии. Чтение из адреса = статус периферии.

Вот как это выглядит «под капотом»:

```c
// В заголовочном файле (stm32f4xx.h):
#define PERIPH_BASE       0x40000000UL
#define AHB1PERIPH_BASE   (PERIPH_BASE + 0x00020000UL)
#define GPIOA_BASE        (AHB1PERIPH_BASE + 0x0000UL)

// Структура регистров GPIO:
typedef struct {
    volatile uint32_t MODER;    // 0x00 — режим (вход/выход/альт.)
    volatile uint32_t OTYPER;   // 0x04 — тип выхода (push-pull/open-drain)
    volatile uint32_t OSPEEDR;  // 0x0C — скорость
    volatile uint32_t PUPDR;    // 0x10 — pull-up/pull-down
    volatile uint32_t IDR;      // 0x12 — входные данные (чтение)
    volatile uint32_t ODR;      // 0x14 — выходные данные (запись)
    volatile uint32_t BSRR;     // 0x18 — атомарная установка/сброс
    volatile uint32_t LCKR;     // 0x1C — блокировка
    volatile uint32_t AFR[2];   // 0x20, 0x24 — альтернативные функции
} GPIO_TypeDef;

// Макрос для доступа:
#define GPIOA  ((GPIO_TypeDef *)GPIOA_BASE)

// Использование:
GPIOA->MODER |= (1 << 10);  // Записать по адресу 0x40020000 + 0x00
```

**Зачем `volatile`?** Без `volatile` компилятор оптимизирует:

```c
// Без volatile:
GPIOA->ODR = 0x00000020;  // Установить бит 5
GPIOA->ODR = 0x00000000;  // Сбросить бит 5
GPIOA->ODR = 0x00000020;  // Снова установить

// Компилятор (без volatile) видит: «три записи подряд, последняя = 0x20»
// Оптимизация: оставить только последнюю запись!
// Результат: мигание НЕ работает, LED горит постоянно

// С volatile:
// Компилятор обязан выполнить КАЖДУЮ запись, потому что «побочный эффект»
```

> **Правило:** ВСЕ регистры периферии — `volatile`. Всегда. Если вы пишете драйвер — все обращения к аппаратным регистрам через `volatile` указатели.

## 6. Boot процесс: от нажатия кнопки до main()

Один из самых важных вопросов для понимания: что происходит, когда вы включаете микроконтроллер? Откуда он знает, какую программу запускать? Где эта программа находится?

### 6.1. Что такое «Reset» и откуда он берётся

Прежде чем микроконтроллер начнёт работать, он должен пройти **сброс (Reset)**. Сброс — это «обнуление» всех регистров процессора, установка их в начальное состояние. Представьте, что вы перезагружаете компьютер.

**Откуда может прийти сброс:**

| Источник   | Расшифровка          | Когда срабатывает                                                  |
| ---------- | -------------------- | ------------------------------------------------------------------ |
| **Кнопка** | —                    | Вы нажали кнопку «Reset» на плате                                  |
| **POR**    | Power-On Reset       | Питание только что появилось (вы включили устройство)              |
| **WWDG**   | Window Watchdog      | Программа «зависла» и не обновляет «сторожевой таймер» вовремя     |
| **IWDG**   | Independent Watchdog | Программа «зависла» и не обновляет независимый «сторожевой таймер» |

> **Что такое Watchdog?** Представьте, что у вас есть собака, которую нужно кормить каждые 5 секунд. Если вы не покормили — значит, с вами что-то не так, и собака начинает лаять (перезагружает систему). Это механизм защиты от «зависания».

### 6.2. Пошаговый разбор загрузки

Теперь — что происходит после сброса, шаг за шагом:

```
Шаг 1: Произошёл сброс
       │
       │  Процессор «просыпается» и знает одно правило:
       │  «Иди по адресу 0x0000 0000 и читай оттуда»
       │
       │  Но что такое 0x0000 0000? Это зависит от перемычек
       │  BOOT0 и BOOT1 на плате:
       │
       │  ┌──────────────────────────────────────────────┐
       │  │ BOOT0 = 0, BOOT1 = X → Адрес 0x0000 0000    │
       │  │   показывает на Flash (ваша программа)       │
       │  │                                              │
       │  │ BOOT0 = 1, BOOT1 = 0 → Адрес 0x0000 0000    │
       │  │   показывает на System Memory                │
       │  │   (заводской загрузчик ST для прошивки       │
       │  │    через USART/USB)                          │
       │  └──────────────────────────────────────────────┘
       │
       │  Обычно BOOT0 = 0, так что грузимся из Flash.
       │
Шаг 2: Чтение начального значения указателя стека (SP)
       │
       │  По адресу 0x0000 0000 лежит ПЕРВОЕ 32-битное число.
       │  Это — начальное значение SP (Stack Pointer, указатель стека).
       │  Обычно: конец SRAM = 0x20020000 (для STM32F407 с 128 КБ SRAM)
       │
       │  SP загружается в регистр процессора.
       │
Шаг 3: Чтение адреса функции Reset_Handler
       │
       │  По адресу 0x0000 0004 лежит ВТОРОЕ 32-битное число.
       │  Это — адрес функции Reset_Handler (обработчик сброса).
       │  Обычно: 0x08000000 + смещение (начало вашего кода во Flash)
       │
       │  Процессор переходит по этому адресу и начинает выполнять.
       │
Шаг 4: Выполняется Reset_Handler
       │
       │  Это короткая функция на ассемблере, которая делает
       │  три важные вещи (по порядку):
       │
       │  ┌────────────────────────────────────────────────────┐
       │  │ 4a. Копирование секции .data из Flash в SRAM      │
       │  │     (инициализированные переменные)                │
       │  │     int counter = 0;  ← «0» хранится во Flash,    │
       │  │                       копируется в SRAM при старте │
       │  │                                                    │
       │  │ 4b. Обнуление секции .bss в SRAM                  │
       │  │     (неинициализированные глобальные переменные)   │
       │  │     int buffer[1000];  ← не занимает место во     │
       │  │                       Flash, просто обнуляется     │
       │  │                                                    │
       │  │ 4c. Вызов SystemInit()                             │
       │  │     Настройка тактирования: перевод с внутреннего │
       │  │     генератора (HSI, 16 МГц) на внешний кварц     │
       │  │     (HSE, 8 МГц) + PLL (умножитель до 168 МГц)    │
       │  └────────────────────────────────────────────────────┘
       │
       └─ 5d. Вызов main()
            │
Шаг 5: main() → ваша программа, while(1) { ... }
```

**Ключевая идея:** микроконтроллер не «угадывает», где программа. В самом начале Flash лежит таблица с адресами (векторная таблица), и процессор просто читает её и переходит по указанным адресам.

### 6.3. Таблица векторов (Vector Table) — «адресная книга» обработчиков

Теперь подробнее о том, что лежит в начале Flash. Это называется **таблица векторов прерываний** (Vector Table).

Представьте, что это — «адресная книга» процессора. Когда происходит какое-то событие (нажата кнопка, пришли данные по USB, сработал таймер), процессор смотрит в эту книгу: «А какую функцию мне нужно вызвать для этого события?»

```
Адрес во Flash    Что хранится                Когда вызывается
───────────────────────────────────────────────────────────────────
0x0000 0000       Начальный SP                При старте (единожды)
0x0000 0004       Reset_Handler               При сбросе/включении
0x0000 0008       NMI_Handler                 NMI (немаскируемое прерывание)
0x0000 000C       HardFault_Handler           Критическая ошибка процессора
0x0000 0010       MemManage_Handler           Ошибка доступа к памяти
0x0000 0014       BusFault_Handler            Ошибка шины (нет ответа от устройства)
0x0000 0018       UsageFault_Handler          Ошибка инструкции (неверный код)
0x0000 001C       Зарезервировано             —
0x0000 0020       Зарезервировано             —
0x0000 0024       Зарезервировано             —
0x0000 0028       Зарезервировано             —
0x0000 002C       SVC_Handler                 Системный вызов (SVC инструкция)
0x0000 0030       DebugMon_Handler            Отладчик подключён
0x0000 0034       Зарезервировано             —
0x0000 0038       PendSV_Handler              Отложенное прерывание (для RTOS)
0x0000 003C       SysTick_Handler             Системный таймер (каждый 1 мс)
                  ─────────────────
                  16 системных прерываний

0x0000 0040       WWDG_IRQHandler             Сторожевой таймер (оконный)
0x0000 0044       PVD_IRQHandler              Детектор пониженного напряжения
0x0000 0048       TAMP_STAMP_IRQHandler       Детектор вскрытия + таймер времени
...               ...                         ... ещё ~90 внешних прерываний
0x0000 0168       OTG_FS_IRQHandler           USB On-The-Go (Full Speed)
```

**Разберём некоторые обработчики подробнее:**

|Название|Расшифровка|Когда срабатывает|Что делать|
|---|---|---|---|
|**Reset_Handler**|Обработчик сброса|При включении/перезагрузке|Инициализация системы|
|**NMI_Handler**|Non-Maskable Interrupt|Критическая ошибка (часы, питание)|Аварийная остановка|
|**HardFault_Handler**|Жёсткая ошибка|Деление на ноль, выход за пределы, неверный адрес|Записать дамп, перезагрузить|
|**MemManage_Handler**|Ошибка управления памятью|Попытка записать в Read-Only память|Записать ошибку, перезагрузить|
|**BusFault_Handler**|Ошибка шины|Обращение к несуществующему устройству|Записать ошибку, перезагрузить|
|**UsageFault_Handler**|Ошибка использования|Неверная инструкция, невыровненный доступ|Записать ошибку, перезагрузить|
|**SVC_Handler**|Supervisor Call|Программа вызвала SVC-инструкцию|Переключение режима (для RTOS)|
|**PendSV_Handler**|Pending System Call|Отложенный системный вызов|Переключение задач (для RTOS)|
|**SysTick_Handler**|System Tick|Системный таймер (каждый 1 мс)|Счётчик времени, RTOS-тик|

> **Зачем это знать вам?** Когда ваша программа «падает» и вы не понимаете почему — чаще всего срабатывает **HardFault_Handler**. Если вы напишете внутри него код для сохранения диагностической информации (какой адрес вызвал ошибку, какие были значения регистров), вы сможете найти причину бага.

### 6.4. Что такое .data и .bss?

Это две **секции** (области) памяти, которые создаёт компилятор. Давайте разберёмся, зачем они нужны и чем отличаются.

#### Секция .data — инициализированные данные

Это глобальные и статические переменные, которым вы задали **начальное значение**:

```c
int counter = 0;           // Глобальная, инициализирована
static int timeout = 100;  // Статическая, инициализирована
const char msg[] = "Hello"; // Константа (хранится отдельно, во Flash)
```

**Проблема:** Flash — энергонезависимая память. Она хранит вашу программу, когда питание выключено. Но когда микроконтроллер работает, переменные должны быть в **SRAM** (оперативной памяти), потому что SRAM быстрее и позволяет запись.

**Решение:** Компилятор помещает начальные значения `.data` во Flash (чтобы они не потерялись при выключении питания). А при запуске **Reset_Handler копирует их из Flash в SRAM**.

```
Flash:                          SRAM:
┌──────────────┐                ┌──────────────┐
│  Программа   │                │              │
│  (код)       │                │              │
├──────────────┤                ├──────────────┤
│  .data       │  ◄──копируем──►│  .data       │
│  (начальные  │     при        │  (рабочие    │
│   значения)  │     старте     │   переменные)│
└──────────────┘                └──────────────┘
```

#### Секция .bss — неинициализированные данные

Это глобальные и статические переменные, которым вы **НЕ задали начальное значение** (или задали 0):

```c
int buffer[1000];       // Глобальный массив, не инициализирован
static int counter;     // Статическая, не инициализирована
int flags = 0;          // Инициализирована нулём → тоже .bss!
```

**Хитрость:** Зачем хранить нули во Flash? Это трата места! Вместо этого компилятор помещает `.bss` в SRAM, а **Reset_Handler просто обнуляет** эту область при старте (записывает нули быстрее, чем копирует из Flash).

```c
// Псевдокод того, что делает Reset_Handler:

void Reset_Handler(void) {
    // 1. Копирование .data из Flash в SRAM
    uint32_t *src = &__data_flash_start;   // Где лежат начальные значения
    uint32_t *dst = &__data_ram_start;     // Где должны быть в SRAM
    while (dst < &__data_ram_end) {
        *dst++ = *src++;   // Копируем слово за словом
    }

    // 2. Обнуление .bss в SRAM
    dst = &__bss_ram_start;
    while (dst < &__bss_ram_end) {
        *dst++ = 0;        // Записываем нули
    }

    // 3. Теперь можно вызывать main()
    main();
}
```

> **Почему это важно знать?** Потому что глобальные переменные «бесплатными» не бывают. Каждая `int x = 5;` занимает 4 байта во Flash (для начального значения) + 4 байта в SRAM (для рабочей копии). При ограниченной Flash (например, 64 КБ у Blue Pill) это имеет значение. Если вам не нужно начальное значение — пишите `int x;` вместо `int x = 0;`. Это сэкономит 4 байта во Flash.

### 6.5. SystemInit() — настройка тактирования (обзорно)

После сброса микроконтроллер работает на **внутреннем RC-генераторе (HSI)** с частотой 16 МГц. Это «безопасный» режим — генератор всегда работает, даже без внешнего кварца.

Но ваш микроконтроллер **может** работать на 168 МГц! Для этого используется **PLL** (Phase-Locked Loop — фазовая автоподстройка частоты). Это «умножитель»: берёт входную частоту (например, 8 МГц от внешнего кварца) и умножает её.

```
         HSI (16 МГц, внутренний RC-генератор)
          │                    │
          │                    │  Если нет внешнего кварца —
          │                    │  процессор остаётся на 16 МГц
          │                    │
HSE (8 МГц, внешний кварц) ────┤
                               │
                           ┌───┴───┐
                           │ PLL_M │  Делитель: ÷ 8
                           └───┬───┘
                               │ 1 МГц
                           ┌───┴───┐
                           │ PLL_N │  Умножитель: × 336
                           └───┬───┘
                               │ 336 МГц (VCO — управляемый генератор)
                           ┌───┴───┐
               ┌───────────┤ PLL_P │  Делитель: ÷ 2
               │           └───┬───┘
               │               │ 168 МГц → SYSCLK (системная частота)
               │           ┌───┴───┐
               │           │ PLL_Q │  Делитель: ÷ 7
               │           └───┬───┘
               │               │ 48 МГц → USB, RNG, SDIO
               │
    ┌──────────┴──────────┐
    │     SYSCLK          │  168 МГц — «сердцебиение» процессора
    └──────┬──────────┬───┘
           │          │
      ┌────┴───┐  ┌───┴────┐
      │ AHB    │  │ APB1   │  Шины периферии
      │ (÷1)   │  │ (÷4)   │
      │ 168МГц │  │ 42 МГц │
      └────┬───┘  └───┬────┘
           │          │
      ┌────┴───┐  ┌───┴──────────┐
      │ APB2   │  │ USART2, SPI2 │
      │ (÷2)   │  │ I2C1-3,      │
      │ 84 МГц │  │ TIM2-7       │
      │        │  └──────────────┘
      │ USART1 │
      │ SPI1   │
      │ ADC    │
      └────────┘
```

**Что здесь происходит:**

1. **HSI / HSE** — источники тактового сигнала. HSI — внутренний (16 МГц, не очень точный). HSE — внешний кварц (8 МГц, точный).
2. **PLL_M, PLL_N, PLL_P, PLL_Q** — делители и множители. Из 8 МГц делаем 168 МГц для процессора и 48 МГц для USB.
3. **SYSCLK** — системная частота. Это «ритм» процессора.
4. **AHB, APB1, APB2** — шины, к которым подключена периферия. Разные шины — разные частоты.

**Зачем разные частоты?** Высокоскоростная периферия (ADC, SPI1, TIM1) подключена к быстрой шине APB2 (84 МГц). Низкоскоростная (I2C, USART2) — к медленной APB1 (42 МГц). Это экономит энергию и упрощает дизайн.

> **Критически важно:** Без правильной настройки PLL ваш микроконтроллер работает на HSI (16 МГц) вместо 168 МГц. Это **в 10 раз медленнее**! Один из первых вопросов при отладке: «А вы уверены, что тактирование настроено?»

Детально тактирование и код SystemInit() мы разберём в отдельной лекции. Сейчас важно понять **принцип**: микроконтроллер может работать на разной частоте, и это настраивается программно.
