# ЧАСТЬ 1: Регистровая модель Cortex-M

## 1. Регистры — «рабочие ячейки» процессора

Представьте, что процессор — это инженер за рабочим столом. **Регистры** — это то, что лежит у него прямо перед руками на столе. **Память (RAM/Flash)** — это шкафы в другой комнате.

- Достать что-то из регистра — **мгновенно** (1 такт)
- Достать что-то из SRAM — **дольше** (несколько тактов, зависит от шины)
- Достать что-то из Flash — **ещё дольше** (wait states)

Поэтому: чем больше работы делается в регистрах, тем быстрее код. Компилятор старается держать переменные в регистрах как можно дольше и «выгружает» их в память только когда регистры заканчиваются.

### 1.1. Полный набор регистров Cortex-M

У Cortex-M **16 основных регистров общего назначения** (R0-R15) + **системные регистры** (CONTROL, PRIMASK и другие), которые управляют режимом работы процессора.

```
┌────────────────────────────────────────────────────────┐
│           Регистры Cortex-M                             │
├────────────────────────────────────────────────────────┤
│ R0   │ Аргумент 1, возвращаемое значение                │ ← caller-saved
│ R1   │ Аргумент 2, старшая часть 64-бит возврата        │
│ R2   │ Аргумент 3                                        │
│ R3   │ Аргумент 4                                        │
├────────────────────────────────────────────────────────┤
│ R4   │ Сохраняемая переменная (callee-saved)            │
│ R5   │ Сохраняемая переменная                            │
│ R6   │ Сохраняемая переменная                            │
│ R7   │ Сохраняемая переменная                            │
│ R8   │ Сохраняемая переменная                            │ ← callee-saved
│ R9   │ Сохраняемая переменная / Thread ID               │
│ R10  │ Сохраняемая переменная                            │
│ R11  │ Сохраняемая переменная (Frame Pointer)           │
├────────────────────────────────────────────────────────┤
│ R12  │ Временный регистр (IP — Intra-Procedure Call)    │
├────────────────────────────────────────────────────────┤
│ R13  │ SP — Stack Pointer (указатель стека)             │
│ R14  │ LR — Link Register (адрес возврата)              │
│ R15  │ PC — Program Counter (счётчик команд)            │
└────────────────────────────────────────────────────────┘

+ Системные регистры (недоступны обычными инструкциями):
  CONTROL, PRIMASK, BASEPRI, FAULTMASK, xPSR, MSP, PSP
```

Давайте разберём каждую группу детально.

## 2. R0-R3: «быстрые» регистры (caller-saved)

Эти регистры используются для передачи **аргументов в функции** и **возврата результатов**. Они **не сохраняются** автоматически при вызове функций. Если вы положили значение в R0, вызвали другую функцию — после возврата там может быть что угодно.

**Почему так сделано?** Чтобы функции были «легче». Если бы каждая функция обязана была сохранять R0-R3, это добавляло бы 4 операции PUSH/POP на каждый вызов. Вместо этого компилятор сам решает: если значение нужно после вызова — он сохранит его в R4-R11 или в стек.

```c
int add(int a, int b) {
    return a + b;
}

// Вызов: result = add(5, 3)
// Компилятор сгенерирует (упрощённо):
//   MOV  r0, #5       // 1-й аргумент → R0
//   MOV  r1, #3       // 2-й аргумент → R1
//   BL   add          // Branch with Link — вызвать add
//                      // Адрес возврата автоматически → LR
//   // После возврата: результат в R0
//   MOV  r4, r0       // Сохранить результат (если нужен дальше)
```

### 2.1. Правило AAPCS (ARM Architecture Procedure Call Standard)

Это стандарт, который определяет, как функции вызывают друг друга. Без него функции, скомпилированные разными компиляторами, не могли бы работать вместе.

| Регистр | Назначение                         | Сохраняется?     |
|---------|------------------------------------|------------------|
| R0      | 1-й аргумент, возвращаемое значение | ❌ Нет (caller)  |
| R1      | 2-й аргумент, старшая часть 64-бит  | ❌ Нет (caller)  |
| R2      | 3-й аргумент                        | ❌ Нет (caller)  |
| R3      | 4-й аргумент                        | ❌ Нет (caller)  |
| R4-R11  | Локальные переменные                | ✅ Да (callee)   |
| R12     | Временный (IP)                      | ❌ Нет (caller)  |
| R13     | SP (Stack Pointer)                  | ✅ Да            |
| R14     | LR (Link Register)                  | ✅ Да (callee)   |

**Если функция принимает больше 4 аргументов** — пятый и последующие передаются через **стек**:

```c
void many_args(int a, int b, int c, int d, int e, int f);
// a→R0, b→R1, c→R2, d→R3
// e, f → PUSH в стек перед вызовом
```

**64-битные значения** (long long, double) занимают два регистра:

```c
long long big_value = 0x12345678ABCDEF00;
// R0 = 0xABCDEF00 (младшая часть)
// R1 = 0x12345678 (старшая часть)
```

## 3. R4-R11: «сохраняемые» регистры (callee-saved)

Эти регистры **обязаны быть сохранены** функцией, если она их использует. Функция кладёт их в стек в начале, восстанавливает в конце. Это гарантия: если вы положили значение в R4 до вызова функции — оно останется там после возврата.

```c
int x = 10;  // Компилятор может разместить в R4
int y = 20;  // Компилятор может разместить в R5
helper();    // helper() НЕ может изменить R4, R5!
// x и y всё ещё на месте!

// Компилятор сгенерирует:
//   PUSH {r4, r5, lr}     // Сохранить R4, R5, LR в стек (3 слова = 12 байт)
//   MOV  r4, #10
//   MOV  r5, #20
//   BL   helper            // helper может использовать R0-R3, R12 свободно
//                           // но НЕ R4, R5 — они сохранены!
//   ADD  r0, r4, r5       // R0 = R4 + R5 = 30
//   POP  {r4, r5, pc}     // Восстановить R4, R5; POP в PC = return
```

**Почему это важно знать?** Когда вы пишете на чистом C — компилятор всё делает за вас. Но когда пишете **ассемблерные вставки** или **обработчики прерываний** — вы должны сами сохранять регистры. Иначе — тихая порча данных, которую невероятно сложно отловить.

---

## 4. R12: Intra-Procedure-call scratch register

Временный регистр. Используется компилятором для промежуточных вычислений при вызове функций. Вам, как правило, не нужно о нём думать. Его иногда называют **IP** (Intra-Procedure Call scratch).

---

## 5. R13: SP — Stack Pointer (указатель стека)

Один из самых важных регистров. Указывает на **вершину стека** — области памяти, где хранятся:

- Локальные переменные функций
- Адреса возврата
- Сохранённые регистры (R4-R11)
- Аргументы функций (если их > 4)

### 5.1. Стек растёт ВНИЗ

Это важно! Когда вы «push» — SP уменьшается. Когда «pop» — увеличивается.

```
Адрес         Содержимое
────────────────────────────────
0x20020000 ───┐
              │  (свободная SRAM)
              │  Стек растёт вниз ↓
              │
0x2001FFFC ───┼── ◄ SP (вершина стека)
0x2001FFF8 ───┼── сохранённый R4
0x2001FFF4 ───┼── сохранённый R5
0x2001FFF0 ───┼── адрес возврата (LR)
0x2001FFEC ───┼── локальная переменная
              │
0x20000000 ───┘  (начало SRAM)
```

**Почему вниз?** Исторически — чтобы стек и куча (heap) росли навстречу друг другу и эффективнее использовали память. Куча растёт вверх (от начала RAM), стек — вниз (от конца RAM).

### 5.2. Два указателя стека: MSP и PSP

У Cortex-M **ДВА** указателя стека. В каждый момент времени активен только один:

| Режим                    | Регистр | Использование                                    |
|--------------------------|---------|--------------------------------------------------|
| Privileged (привилегированный)   | MSP (Main Stack Pointer)    | Основной режим, обработчики прерываний          |
| Unprivileged (непривилегированный) | PSP (Process Stack Pointer)   | Пользовательские задачи в RTOS                  |

Зачем два? Для **RTOS**. Когда операционная система переключается между задачами, каждая задача имеет свой стек (через PSP), а ядро RTOS и все прерывания работают на MSP. Это изолирует задачи друг от друга: если задача переполнит свой стек — это не сломает ядро.

```
SRAM:
┌─────────────────────┐ 0x20020000 (конец)
│    Задача 1 (PSP₁)  │  ← Стек задачи 1
├─────────────────────┤
│    Задача 2 (PSP₂)  │  ← Стек задачи 2
├─────────────────────┤
│    Задача 3 (PSP₃)  │  ← Стек задачи 3
├─────────────────────┤
│  Куча (heap) ↑      │
│  ...                │
│  Глобальные данные   │
│  .bss, .data        │
├─────────────────────┤ 0x20000000 (начало)
│   Ядро RTOS (MSP)   │  ← Стек ядра и ВСЕХ прерываний
└─────────────────────┘
```

На bare-metal (без ОС) используется только **MSP**. PSP не задействован.

## 6. R14: LR — Link Register (регистр связи)

Содержит **адрес возврата** из функции. Когда вы вызываете функцию инструкцией `BL` (Branch with Link), процессор **автоматически** записывает в LR адрес следующей инструкции — туда, куда нужно вернуться.

```c
void foo(void) {
    // LR содержит адрес, куда вернуться после BX lr
}

int main(void) {
    foo();  // BL foo → LR = адрес инструкции после BL
    // ← Сюда вернётся foo() через BX lr
}
```

### 6.1. Вложенные вызовы

При вложенных вызовах LR **перезаписывается**. Поэтому если `main()` → `foo()` → `bar()`, то:

- Внутри `foo()`: LR = адрес возврата в `main()`
- Внутри `bar()`: LR = адрес возврата в `foo()` (а адрес в `main()` потерян, если `foo()` не сохранил!)

```c
void bar(void) {
    // LR = адрес возврата в foo()
    BX lr  // return в foo()
}

void foo(void) {
    // LR = адрес возврата в main()
    PUSH {lr}       // СОХРАНИТЬ LR в стек!
    bar();          // BL bar → LR перезаписан адресом возврата из bar
    POP {lr}        // Восстановить LR = адрес в main()
    BX lr           // return в main()
}
```

### 6.2. Особые значения LR

При входе в **обработчик прерывания** LR получает специальное значение, которое говорит процессору, куда вернуться:

| Значение LR    | Значение                                           |
|----------------|----------------------------------------------------|
| 0xFFFFFFF9     | Вернуться в Thread mode, использовать MSP         |
| 0xFFFFFFFD     | Вернуться в Thread mode, использовать PSP         |
| 0xFFFFFFF1     | Вернуться в Handler mode, использовать MSP        |

Это используется автоматически — вам не нужно об этом заботиться. Но полезно знать для отладки: если вы видите в LR значение `0xFFFFFFF9`, значит вы находитесь внутри обработчика прерывания.

## 7. R15: PC — Program Counter (счётчик команд)

Содержит адрес **текущей выполняемой инструкции**. Процессор читает инструкцию по адресу в PC, выполняет, увеличивает PC на размер инструкции, читает следующую.

**Интересный факт:** из-за конвейеризации (pipeline) в PC хранится адрес инструкции **на 2 шага вперёд**. Если вы читаете PC в коде, там будет `текущий_адрес + 4` (для 32-битных инструкций Thumb-2).

```c
uint32_t get_pc(void) {
    // Инструкция: MOV r0, pc
    // Вернёт: адрес этой инструкции + 4
    __asm volatile("MOV r0, pc");
}
```

## 8. xPSR — Program Status Register

Это 32-битный регистр состояния, состоящий из трёх частей:

```
 31                                  0
┌──────────────┬─────────┬──────────────────┐
│    APSR      │  IPSR   │     EPSR         │
│ (флаги)      │(номер   │ (управление       │
│ 31..28       │прерыв.) │  выполнением)     │
│              │ 9..0    │ 27..10            │
└──────────────┴─────────┴──────────────────┘
```

### 8.1. APSR (Application PSR) — флаги состояния

Четыре флага, которые устанавливаются после арифметических и логических операций:

| Бит | Флаг | Название   | Когда устанавливается                              |
|-----|------|------------|-----------------------------------------------------|
| 31  | N    | Negative   | Результат операции отрицательный (старший бит = 1)   |
| 30  | Z    | Zero       | Результат равен нулю                                |
| 29  | C    | Carry      | Был перенос/заём (для беззнаковых операций)          |
| 28  | V    | Overflow   | Переполнение (для знаковых операций)                 |

Эти флаги — основа **условных переходов**:

```c
// if (a == b) { ... }
// Компилятор:
//   CMP  r0, r1       // Сравнить: R0 - R1, результат НЕ сохраняется, только флаги
//   BEQ  equal_label  // Branch if Equal — перейти если Z=1

// if (a > b) { ... }  (знаковое сравнение)
//   CMP  r0, r1
//   BGT  greater_label  // Branch if Greater Than: Z=0 И N=V
```

**Пример ручной проверки:**

```c
int a = 5, b = 5;
// CMP r0, r1 → 5 - 5 = 0 → Z=1, N=0, C=1, V=0
// BEQ → Z=1 → переход выполнится

int c = 10, d = 20;
// CMP r0, r1 → 10 - 20 = -10 → Z=0, N=1, C=0, V=0
// BGT → Z=0 И N≠V → N=1, V=0 → не равно → переход НЕ выполнится
```

### 8.2. IPSR (Interrupt PSR) — номер текущего прерывания

- **0** = не в прерывании (Thread mode, обычный код)
- **1** = NMI (Non-Maskable Interrupt)
- **2** = HardFault
- **3** = MemManage
- **4** = BusFault
- **5** = UsageFault
- **15** = SysTick
- **16+** = внешние прерывания (IRQ 0, IRQ 1, …)

Полезно для отладки:

```c
void HardFault_Handler(void) {
    // Узнать, какое прерывание было активно до HardFault
    volatile uint32_t irq_num = (__get_IPSR() & 0x1FF);
    // Если irq_num == 0 — HardFault случился в main(), не в прерывании
    // Если irq_num > 0 — HardFault случился внутри обработчика прерывания №irq_num
}
```

### 8.3. EPSR (Execution PSR) — управление выполнением

В основном используется internally процессором. Один бит важен для отладки:

| Бит | Название   | Значение                                    |
|-----|------------|---------------------------------------------|
| 24  | T (Thumb)  | Всегда 1 на Cortex-M (только Thumb-инструкции) |

## 9. Системные (специальные) регистры

Помимо R0-R15 и xPSR, есть регистры, которые управляют **режимом работы** процессора. Они **не доступны** обычными инструкциями MOV — нужны специальные:

| Инструкция | Описание                               |
|------------|----------------------------------------|
| `MRS`      | Move to Register from Special (прочитать системный регистр) |
| `MSR`      | Move to Register from Special (записать в системный регистр) |

В C эти операции оборачиваются функциями CMSIS: `__get_CONTROL()`, `__set_BASEPRI()` и т.д.

### 9.1. CONTROL — регистр управления режимами

```
 31                              2       1       0
┌──────────────────────────────┬───────┬───────┬───────┐
│        Reserved              │ FPCA  │ SPSEL │ nPRIV │
└──────────────────────────────┴───────┴───────┴───────┘
```

| Бит | Название | 0                    | 1                          |
|-----|----------|----------------------|----------------------------|
| 0   | nPRIV    | Privileged режим     | Unprivileged режим         |
| 1   | SPSEL    | Использовать MSP     | Использовать PSP           |
| 2   | FPCA     | FPU не активен       | FPU активен (контекст сохраняется) |

**nPRIV — уровень привилегий:**

- **Privileged (nPRIV=0):** полный доступ ко всем инструкциям и регистрам. Можно писать в CONTROL, менять режимы, настраивать NVIC.
- **Unprivileged (nPRIV=1):** ограниченный доступ. Нельзя писать в системные регистры, нельзя менять CONTROL, нельзя отключать прерывания. Если попытаться — будет **UsageFault**.

**Зачем?** Для защиты. В RTOS пользовательские задачи работают в unprivileged режиме. Если задача «сломается» и попытается записать в системный регистр — будет исключение, а не падение всей системы.

Переход из privileged в unprivileged — **односторонняя дверь**. Вернуться можно только через прерывание **SVC** (Supervisor Call).

```c
// Переход в unprivileged режим
__set_CONTROL(__get_CONTROL() | 1);  // nPRIV = 1
// Теперь нельзя писать в системные регистры!
// __disable_irq();  // → UsageFault!
// Чтобы вернуться — только через SVC-вызов
```

### 9.2. PRIMASK — глобальная маска прерываний

Однобитный регистр. Когда PRIMASK=1 — **все прерывания с настраиваемым приоритетом заблокированы**.

**НЕ маскирует:** NMI и HardFault (они всегда работают, независимо от PRIMASK).

```c
// Критическая секция — атомарная операция
__disable_irq();    // PRIMASK = 1, прерывания заблокированы
// ... критический код (несколько инструкций) ...
__enable_irq();     // PRIMASK = 0, прерывания разрешены
```

**Когда использовать:**
- Короткие критические секции (несколько инструкций)
- Защита разделяемых данных между main() и ISR
- Атомарные операции, когда нет аппаратной поддержки

**Когда НЕ использовать:**
- Длинные участки кода (прерывания будут копиться, потом все сработают сразу — «interrupt storm»)
- В RTOS (заблокирует шедулер)
- Когда важна обработка критических событий (watchdog, аварийная остановка)

### 9.3. BASEPRI — пороговая маска прерываний (только M3/M4/M7)

Более «умная» маска. Блокирует только прерывания с приоритетом **ниже** заданного порога. Прерывания с более высоким приоритетом (меньшим числом) продолжат работать.

```c
// Приоритеты в STM32F4: 0 = высший, 15 = низший (4 бита)
// BASEPRI сравнивает с 8 битами, поэтому приоритет нужно сдвинуть влево на 4

// Заблокировать все прерывания с приоритетом >= 128 (0x80)
// То есть приоритеты 8-15 (0x80-0xF0) будут заблокированы
// А приоритеты 0-7 (0x00-0x70) — продолжат работать!
__set_BASEPRI(0x80);

// ... код ...

// Сбросить — все прерывания снова работают
__set_BASEPRI(0x00);
```

**Почему это лучше PRIMASK в RTOS?** Вы можете заблокировать обычные прерывания, но оставить работающими критические (watchdog, аварийная остановка мотора).

> **Промышленный стандарт:** В RTOS-приложениях используйте `BASEPRI` для критических секций. FreeRTOS использует `BASEPRI` для `taskENTER_CRITICAL()`. PRIMASK — только для экстренных случаев.

### 9.4. FAULTMASK — маска всех исключений

Ещё более «грубая», чем PRIMASK. Маскирует **ВСЕ** исключения, **включая HardFault**. Только NMI остаётся.

```c
__disable_fault_irq();  // FAULTMASK = 1
// Даже HardFault не сработает!
__enable_fault_irq();   // FAULTMASK = 0
```

На практике используется крайне редко. Если вы его используете — у вас, скорее всего, уже есть серьёзная проблема в архитектуре.

### 9.5. Сводная таблица масок прерываний

| Регистр    | Блокирует                     | НЕ блокирует           | Когда использовать                  |
|------------|-------------------------------|-------------------------|-------------------------------------|
| PRIMASK    | Все IRQ с настраиваемым приоритетом | NMI, HardFault     | Короткие критические секции         |
| BASEPRI    | IRQ с приоритетом ниже порога | IRQ выше порога, NMI, HardFault | RTOS-критические секции |
| FAULTMASK  | ВСЕ исключения                | NMI                     | Почти никогда (только экстренные случаи) |

## 10. NVIC — контроллер прерываний

**NVIC (Nested Vectored Interrupt Controller)** — это «диспетчер» прерываний, встроенный в ядро Cortex-M. Он решает:

1. Какое прерывание обслужить сейчас?
2. Можно ли прервать текущий обработчик более приоритетным?
3. Какие прерывания замаскировать?

### 10.1. Ключевые характеристики NVIC

| Параметр         | Значение                                                    |
|------------------|--------------------------------------------------------------|
| Прерывания       | До 240 внешних (IRQ) + 16 системных                         |
| Приоритеты       | До 256 уровней (8 бит). В STM32F4 — **16 уровней** (4 бита) |
| Вложенность      | Да, прерывание может прервать другое прерывание             |
| Время отклика    | **12 тактов** (от запроса до первой инструкции обработчика)  |
| Tail-chaining    | Два ожидающих прерывания → переход за 6 тактов (без сохранения/восстановления контекста) |
| Late-arriving    | Пришло более приоритетное во время входа → переключается автоматически |

## 11. SysTick — системный таймер ядра

**SysTick** — простой 24-битный таймер обратного отсчёта, встроенный в **ядро** Cortex-M (не в периферию STM32). Есть на **любом** Cortex-M, независимо от производителя.

Зачем он нужен:

1. **Тик RTOS** — FreeRTOS использует SysTick для переключения задач (обычно 100-1000 Гц)
2. **Задержки** — `HAL_Delay()` работает через SysTick
3. **Таймауты** — отследить, что операция «зависла»

```c
// Настройка SysTick на 1 мс (при SYSCLK 168 МГц)
// SysTick_Config(168000) → 168 000 000 / 1000 = 168 000 тактов
SysTick_Config(168000);

// Обработчик (определён в вашем коде)
void SysTick_Handler(void) {
    // Вызывается каждые 1 мс
    HAL_IncTick();  // Увеличивает внутренний счётчик HAL
}

// Использование
HAL_Delay(100);  // Ждать 100 мс (100 тиков SysTick)
```

# ЧАСТЬ 2: Система тактирования STM32 (RCC)

Теперь переходим от ядра к периферии. Без правильной настройки тактирования **ни один блок периферии не заработает**. Это одна из самых частых ошибок новичков: «Я настроил GPIO, а LED не мигает!» — «А тактирование включили?»

## 12. Зачем нужна система тактирования?

### 12.1. Аналогия с сердцем

Представьте, что микроконтроллер — это организм. **Ядро (Cortex-M)** — это мозг. **Периферия (GPIO, USART, SPI, ADC, TIM)** — это органы: руки, ноги, глаза. А **система тактирования (RCC)** — это сердце, которое качает кровь (тактовые импульсы) по всему организму.

Если сердце не качает кровь к руке — рука «мертвая». Она есть, но не двигается. Точно так же: если RCC не подаёт тактовый сигнал на GPIO — регистры GPIO не отвечают. Запись в них — как крик в пустоту.

### 12.2. Почему не тактировать всё сразу?

Казалось бы — почему бы не подать тактовый сигнал на всю периферию сразу? Причина — **энергопотребление**. Каждый блок периферии, даже если он не используется, потребляет ток, когда получает тактовый сигнал. В мобильных и battery-powered устройствах это критично.

Поэтому после сброса **вся периферия отключена**. Вы явно включаете только то, что используете:

```c
// БЕЗ этого — GPIOG не работает:
RCC->AHB1ENR |= RCC_AHB1ENR_GPIOGEN;  // Включить тактирование GPIOG

// Теперь можно:
GPIOG->MODER |= GPIO_MODER_MODE13_0;  // PG13 — Output
```

## 13. Источники тактового сигнала

У STM32F4 есть **четыре** источника тактового сигнала:

| Источник | Полное название       | Частота            | Расположение     | Точность  | Работает в Stop mode |
|----------|-----------------------|--------------------|------------------|-----------|----------------------|
| **HSI**  | High Speed Internal   | 16 МГц             | Внутри чипа      | ±1%       | ✅ Да                 |
| **HSE**  | High Speed External   | 4-26 МГц (обычно 8 МГц) | Внешний кварц на плате | ±50 ppm | ❌ Нет (зависит от схемы) |
| **LSI**  | Low Speed Internal    | ~32 кГц            | Внутри чипа      | ±40%      | ✅ Да                 |
| **LSE**  | Low Speed External    | 32 768 Гц          | Внешний кварц    | ±20 ppm   | ✅ Да                 |

Давайте разберём каждый подробно.

### 13.1. HSI — High Speed Internal (внутренний быстрый генератор)

**Что это:** Встроенный RC-генератор на **16 МГц** (для STM32F4). Работает всегда, даже сразу после включения, до настройки чего-либо.

**Плюсы:**
- Не требует внешних компонентов
- Запускается мгновенно (~2 мкс)
- Работает в режимах低功耗 (Stop, Sleep)

**Минусы:**
- Неточный: ±1% (зависит от температуры и напряжения)
- Для USART это может быть критично (погрешность baud rate > 2% = ошибки связи)
- Для USB — неприемлем (нужна точность ±0.25%)

**Когда использовать:**
- На этапе отладки, пока не настроен HSE+PLL
- В энергоэффективных приложениях, где точность не важна
- Как резервный источник (если HSE отказал)

### 13.2. HSE — High Speed External (внешний быстрый генератор)

**Что это:** Внешний кварцевый резонатор, подключённый к пинам **OSC_IN** и **OSC_OUT** на плате.

На **Nucleo-F401RE**: HSE = **8 МГц** (от ST-Link генератора)
На **F429I-DISC1**: HSE = **8 МГц** (отдельный кварц)

**Плюсы:**
- Точный: ±50 ppm (в 200 раз точнее HSI!)
- Стабильный (не плывёт от температуры)
- Нужен для USB, SDIO, Ethernet

**Минусы:**
- Требует внешнего кварца (занимает место на плате)
- Запускается медленнее (~2-5 мс)
- Может быть отключён в Stop mode (зависит от схемы)

**Почему 8 МГц?** Это стандартное значение. 8 МГц удобно делится и умножается до нужных частот. Некоторые платы используют 12 МГц, 25 МГц — это влияет на расчёт PLL.

### 13.3. LSI — Low Speed Internal (внутренний медленный генератор)

**Что это:** Встроенный RC-генератор на **~32 кГц**. Очень неточный (±40%!), но работает всегда, даже в Stop mode.

**Используется для:**
- **IWDG** (Independent Watchdog) — сторожевой таймер ДОЛЖЕН работать даже в Stop mode
- **RTC** (если нет LSE) — но часы будут «плыть»

**Плюсы:**
- Работает всегда, без внешних компонентов
- Потребляет ничтожный ток

**Минусы:**
- Частота «плавает» от 20 кГц до 45 кГц (±40%!)
- Для точных часов неприемлем

### 13.4. LSE — Low Speed External (внешний медленный генератор)

**Что это:** Внешний часовой кварц на **32 768 Гц** (2¹⁵ = 32768 — выбрано не случайно).

**Почему 32 768?** Это число удобно делится нацело: 32768 → 16384 → 8192 → ... → 1 Гц. Ровно один тик в секунду. Идеально для часов.

**Используется для:**
- **RTC** — точные часы (погрешность ~20 секунд в год)
- **RTC Wakeup Timer** — пробуждение из Stop mode по расписанию

**Плюсы:**
- Очень точный: ±20 ppm (~1 секунда в месяц)
- Очень низкое потребление

**Минусы:**
- Требует внешнего кварца
- На Nucleo-F401RE LSE **не распаян** по умолчанию (нужно паять самостоятельно)
- На F429I-DISC1 LSE **распаян**

## 14. Блок PLL — умножение частоты

Теперь — ключевая часть. HSE = 8 МГц. Но процессор STM32F4 хочет работать на **168 МГц**. Как из 8 получить 168? Через **PLL** (Phase-Locked Loop — фазовая автоподстройка частоты).

### 14.1. Что такое PLL?

PLL — это «умножитель» частоты. Он берёт входную частоту, умножает её на коэффициент, делит — и получает нужную выходную.

**Простая аналогия:** представьте, что у вас есть велосипед с передачами. Вы крутите педали 1 раз (входная частота), а колесо крутится 21 раз (выходная частота). Передаточное число — это PLL-коэффициенты.

### 14.2. Структура PLL в STM32F4

У STM32F4 есть **один основной PLL** с четырьмя настраиваемыми параметрами:

```
                    PLLM         PLLN        PLLP / PLLQ
                   (делитель)  (множитель)  (делитель)
                     ↓            ↓            ↓
HSE/HSE ──►  ┌─────┐      ┌──────────┐    ┌───────┐
8 МГц        │ ÷ M │──1──►│  × N     │───►│ ÷ P   │──► SYSCLK (168 МГц)
             │     │  МГц │  (VCO)   │    │       │
             └─────┘      │ 100-432  │    └───────┘
                          │  МГц     │       │
                          └──────────┘    ┌───────┐
                                          │ ÷ Q   │──► 48 МГц (USB, RNG)
                                          │       │
                                          └───────┘
```

Четыре параметра:

| Параметр | Что делает             | Диапазон   | Для чего             |
|----------|------------------------|------------|-----------------------|
| **PLLM** | Делит входную частоту  | 2-63       | Чтобы получить 1-2 МГц на входе VCO |
| **PLLN** | Умножает (VCO)         | 50-432     | Основная частота VCO (100-432 МГц) |
| **PLLP** | Делит для SYSCLK       | 2, 4, 6, 8 | Системная частота процессора |
| **PLLQ** | Делит для USB/RNG/SDIO | 2-15       | 48 МГц для USB        |

### 14.3. Расчёт SYSCLK — формула

```
                HSE (или HSI)
SYSCLK = ───────────────────── × PLLN ÷ PLLP
              PLLM
```

**Пример для STM32F407 (HSE = 8 МГц, SYSCLK = 168 МГц):**

```
PLLM = 8     →  8 МГц ÷ 8   = 1 МГц (вход VCO)
PLLN = 336   →  1 МГц × 336 = 336 МГц (VCO)
PLLP = 2     →  336 МГц ÷ 2 = 168 МГц (SYSCLK) ✅

PLLQ = 7     →  336 МГц ÷ 7 = 48 МГц (USB) ✅
```

**Пример для Nucleo-F401RE (HSE = 8 МГц, SYSCLK = 84 МГц):**

```
PLLM = 8     →  8 МГц ÷ 8   = 1 МГц (вход VCO)
PLLN = 336   →  1 МГц × 336 = 336 МГц (VCO)
PLLP = 4     →  336 МГц ÷ 4 = 84 МГц (SYSCLK) ✅

PLLQ = 7     →  336 МГц ÷ 7 = 48 МГц (USB, но на Nucleo USB не используется)
```

### 14.4. Важные ограничения PLL

При настройке PLL нужно соблюдать **три критических ограничения**:

| Параметр            | Ограничение              | Почему                                    |
|---------------------|--------------------------|-------------------------------------------|
| Вход VCO (после PLLM) | **1-2 МГц**             | VCO работает только в этом диапазоне       |
| Частота VCO (после PLLN) | **100-432 МГц** (STM32F4) | Аппаратное ограничение VCO              |
| SYSCLK (после PLLP)    | **макс 168 МГц** (STM32F4) | Максимальная частота ядра                |

Если нарушить хоть одно — PLL не заблокируется (HSE_READY не установится) или процессор будет работать нестабильно.

**Проверка для нашего примера:**
- Вход VCO: 8 МГц ÷ 8 = 1 МГц ✅ (в диапазоне 1-2 МГц)
- VCO: 1 МГц × 336 = 336 МГц ✅ (в диапазоне 100-432 МГц)
- SYSCLK: 336 ÷ 2 = 168 МГц ✅ (не превышает максимум)

## 15. Шины: AHB, APB1, APB2

SYSCLK — это частота процессора. Но периферия работает на **своих** частотах. Они получаются из SYSCLK через делители.

### 15.1. Три шины

```
                    SYSCLK (168 МГц)
                         │
                    ┌────┴────┐
                    │  AHB    │  делитель HPRE (обычно ÷1)
                    │  168 МГц│
                    └────┬────┘
                         │
              ┌──────────┼──────────┐
              ▼          ▼          ▼
         ┌────────┐  ┌────────┐  ┌────────┐
         │  APB1  │  │  APB2  │  │  AHB1  │
         │ ÷ 4    │  │ ÷ 2    │  │ ÷ 1    │
         │ 42 МГц │  │ 84 МГц │  │ 168 МГц│
         └────────┘  └────────┘  └────────┘
```

| Шина | Полное название     | Макс частота | Делитель от HCLK | Периферия                                          |
|------|---------------------|--------------|-------------------|----------------------------------------------------|
| **AHB**  | Advanced High-performance Bus | 168 МГц | ÷1 (HPRE) | DMA1-2, Flash, RCC, CRC, FSMC, Crypto |
| **APB1** | APB Low-speed    | 42 МГц  | ÷4 (PPRE1) | USART2-3, SPI2-3, I2C1-3, TIM2-7, WWDG, PWR, DAC |
| **APB2** | APB High-speed   | 84 МГц  | ÷2 (PPRE2) | USART1, USART6, SPI1, ADC1-3, TIM1, TIM8, SYSCFG, все GPIO |

**Почему разделение?** Не всей периферии нужна высокая частота. I2C работает максимум на 400 кбит/с (Fast mode) — зачем ему 168 МГц? USART1 может на 10.5 Мбит/с — нужна быстрая шина. ADC — до 36 МГц — тоже быстрая шина.

### 15.2. Важный нюанс: тактирование таймеров

Частота таймеров (TIM) — **не всегда равна** частоте шины. Если делитель APB ≠ 1, частота таймера **удваивается**:

```
APB1 = 42 МГц (делитель ÷4 от HCLK)
→ TIM2-7 тактируются на 84 МГц (×2!)

APB2 = 84 МГц (делитель ÷2 от HCLK)
→ TIM1, TIM8 тактируются на 168 МГц (×2!)
```

Это частая ловушка! Если вы настраиваете TIM2 и думаете, что он работает на 42 МГц — вы получите неправильную частоту ШИМ.

```c
// TIM2 на APB1
// APB1 = 42 МГц, но TIM2 тактируется на 84 МГц (×2)
// Для PWM на 1 кГц:
// PSC = 84 000 000 / 1 000 - 1 = 83 999
// ARR = 1000 - 1 = 999 (для 1 кГц PWM)
// Реальная частота: 84 000 000 / (84000 × 1000) = 1 Гц... 
// Подождите, давайте правильно:
// Частота PWM = 84 000 000 / (PSC+1) / (ARR+1)
// Для 1 кГц: PSC = 83, ARR = 999 → 84 000 000 / 84 / 1000 = 1000 Гц ✅
```

### 15.3. Полная схема тактирования STM32F4

Вот как выглядит вся система целиком:

```
┌──────────┐   ┌──────────┐
│  HSI     │   │  HSE     │
│  16 МГц  │   │  8 МГц   │
└────┬─────┘   └────┬─────┘
     │              │
     │    ┌─────────┘
     │    │
     ▼    ▼
  ┌──────────────┐
  │  Выбор SYS   │  ◄── RCC_CFGR: биты SW[1:0]
  │  (SYSCLK)    │      00 = HSI, 01 = HSE, 10 = PLL
  └──────┬───────┘
         │
         ▼
  ┌──────────────┐     ┌──────────────────────────┐
  │    PLL       │     │  CSS (Clock Security     │
  │  (умножитель)│     │   System)                │
  └──────┬───────┘     │  Если HSE отказал →      │
         │             │  автоматически HSI       │
         ▼             └──────────────────────────┘
     SYSCLK (168 МГц)
         │
    ┌────┴────┐
    │  HPRE   │  AHB prescaler (÷1)
    │  ÷ 1    │
    └────┬────┘
         │
       HCLK (168 МГц) ──► Ядро, DMA, Flash, FSMC
         │

    │                   │

└───┬────┘        └─────┬────┘
    │                   │
PCLK1 (42 МГц)    PCLK2 (84 МГц)
    │                   │
    ▼                   ▼
┌────────┐         ┌────────┐
│ APB1   │         │  APB2  │
│USART2-3│         │USART1,6│
│SPI2-3  │         │SPI1    │

│TIM2-7  │×2=84МГц│TIM1,8  │×2=168МГц

│DAC     │         │все GPIO│
└────────┘         └────────┘
```

## 16. Тактирование периферии — регистры RCC_EN

Каждый блок периферии имеет свой бит включения в регистрах RCC. До установки этого бита регистры периферии **не отвечают**.

### 16.1. Три регистра включения

| Регистр         | Шина  | Что включает                              |
|-----------------|-------|--------------------------------------------|
| **RCC_AHB1ENR**  | AHB1  | GPIOA-I, DMA1-2, OTGHS, ETH, CRC, Flash   |
| **RCC_APB1ENR**  | APB1  | USART2-3, SPI2-3, I2C1-3, TIM2-7, WWDG, PWR, DAC |
| **RCC_APB2ENR**  | APB2  | USART1, USART6, SPI1, ADC1-3, TIM1, TIM8, SYSCFG, все GPIO |

### 16.2. Примеры включения периферии

```c
// Включить тактирование GPIOA, GPIOG (AHB1)
RCC->AHB1ENR |= RCC_AHB1ENR_GPIOAEN | RCC_AHB1ENR_GPIOGEN;

// Включить тактирование USART2 (APB1)
RCC->APB1ENR |= RCC_APB1ENR_USART2EN;

// Включить тактирование SPI1 (APB2)
RCC->APB2ENR |= RCC_APB2ENR_SPI1EN;

// Включить тактирование TIM2 (APB1)
RCC->APB1ENR |= RCC_APB1ENR_TIM2EN;

// Включить тактирование ADC1 (APB2)
RCC->APB2ENR |= RCC_APB2ENR_ADC1EN;
```

> **Золотое правило:** ПЕРВЫМ делом — RCC_EN. ВТОРЫМ — настройка регистров периферии. Если вы попробуете настроить GPIO до включения RCC_AHB1ENR — запись «пролетит в пустоту». Регистр не примет значение. И никаких ошибок вы не получите — просто ничего не произойдёт.

### 16.3. Как узнать, какой регистр EN нужен?

Откройте **Reference Manual** (RM0090 для STM32F4), раздел «RCC». Там есть таблица:

```
Периферия   │ Шина  │ Регистр включения
────────────┼───────┼──────────────────
GPIOA       │ AHB1  │ RCC_AHB1ENR, бит 0
GPIOB       │ AHB1  │ RCC_AHB1ENR, бит 1
...
GPIOG       │ AHB1  │ RCC_AHB1ENR, бит 6
USART1      │ APB2  │ RCC_APB2ENR, бит 4
USART2      │ APB1  │ RCC_APB1ENR, бит 17
SPI1        │ APB2  │ RCC_APB2ENR, бит 12
TIM2        │ APB1  │ RCC_APB1ENR, бит 0
ADC1        │ APB2  │ RCC_APB2ENR, бит 8
```

## 17. MCO — вывод тактового сигнала на пин

**MCO** (Master Clock Output) — позволяет вывести тактовый сигнал на пин **PA8**. Это полезно для:

1. Проверки осциллографом: «А действительно ли SYSCLK = 168 МГц?»
2. Тактирования внешних устройств (других чипов на плате)
3. Отладки: если MCP показывает 16 МГц вместо 168 — PLL не настроен

### 17.1. MCO1 (PA8) и MCO2 (PC9)

У STM32F4 два вывода MCO:

| Вывод | Пин | Доступные источники                          |
|-------|-----|-----------------------------------------------|
| MCO1  | PA8 | HSI, LSE, HSE, PLLCLK (SYSCLK)               |
| MCO2  | PC9 | SYSCLK, PLLI2S, HSE, PLLCLK                  |

### 17.2. Настройка MCO1

```c
// 1. Включаем тактирование GPIOA
RCC->AHB1ENR |= RCC_AHB1ENR_GPIOAEN;

// 2. PA8 — альтернативная функция MCO1 (AF0)
GPIOA->MODER &= ~GPIO_MODER_MODE8;
GPIOA->MODER |= GPIO_MODER_MODE8_1;  // AF (10)
GPIOA->OSPEEDR |= GPIO_OSPEEDER_OSPEED8;  // Very High (для 168 МГц!)

// 3. Настраиваем MCO1: источник + делитель
RCC->CFGR &= ~(RCC_CFGR_MCO1 | RCC_CFGR_MCO1PRE);
RCC->CFGR |= RCC_CFGR_MCO1_PLLCLK;      // Источник = PLLCLK (SYSCLK)
RCC->CFGR |= RCC_CFGR_MCO1PRE_DIV2;     // Делитель = ÷2 (168 ÷ 2 = 84 МГц)

// На PA8 теперь — меандр 84 МГц (максимум для пинов STM32F4)
```

> **Важно:** Максимальная частота на пине GPIO — **~84 МГц**. Поэтому SYSCLK (168 МГц) нужно делить минимум на 2.

## 18. CSS — Clock Security System (защита от отказа часов)

**CSS** — аппаратный механизм, который мониторит HSE. Если HSE «упал» (кварц перестал генерировать) — CSS автоматически:

1. Переключает SYSCLK на HSI (16 МГц)
2. Генерирует прерывание **NMI** (Non-Maskable Interrupt)

Вы не можете отключить CSS после включения. Это одноразовый «предохранитель».

```c
// Включить CSS
RCC->CR |= RCC_CR_CSSON;

// Если HSE отказал — вызовется NMI_Handler
void NMI_Handler(void) {
    // Проверить, что это CSS NMI (а не другой NMI)
    if (__HAL_RCC_GET_FLAG(RCC_FLAG_CSS) != RESET) {
        // HSE отказал! Мы теперь на HSI (16 МГц)
        // Нужно: понизить baud rate USART, перезапустить HSE, записать лог
    }
}
```

**Зачем это нужно?** Представьте устройство управления мотором. Если кварц «упал» — процессор перестал работать. Мотор продолжает крутиться на последней команде. Это может привести к поломке. CSS переключает на HSI — пусть медленнее, но устройство хотя бы сможет остановить мотор безопасно.

## 19. Пошаговая настройка тактирования (полный пример)

Вот полный код настройки SYSCLK 168 МГц на STM32F407:

```c
#include "stm32f4xx.h"

void SystemClock_Config(void) {
    // ── Шаг 1: Включить HSE ──
    RCC->CR |= RCC_CR_HSEON;              // Запустить HSE
    while (!(RCC->CR & RCC_CR_HSERDY));   // Ждать готовности (HSE Ready)

    // ── Шаг 2: Настроить Flash latency ──
    // При 168 МГц нужен FLASH_LATENCY_5 (6 тактов чтения)
    // Если оставить LATENCY_0 — HardFault при обращении к Flash!
    FLASH->ACR = FLASH_ACR_PRFTEN         // Prefetch buffer
               | FLASH_ACR_ICEN            // Instruction cache
               | FLASH_ACR_DCEN            // Data cache
               | FLASH_ACR_LATENCY_5WS;    // 5 wait states

    // ── Шаг 3: Настроить делители шин ──
    RCC->CFGR |= RCC_CFGR_PPRE2_DIV2;     // APB2 = HCLK/2 = 84 МГц
    RCC->CFGR |= RCC_CFGR_PPRE1_DIV4;     // APB1 = HCLK/4 = 42 МГц
    // AHB делитель = 1 (по умолчанию), HCLK = SYSCLK = 168 МГц

    // ── Шаг 4: Настроить PLL ──
    // HSE = 8 МГц
    RCC->PLLCFGR = (8)                    // PLLM = 8  → 8/8 = 1 МГц
                 | (336 << 6)             // PLLN = 336 → 1 × 336 = 336 МГц (VCO)
                 | (0 << 16)              // PLLP = 2  → 336/2 = 168 МГц (SYSCLK)
                 | (7 << 24)              // PLLQ = 7  → 336/7 = 48 МГц (USB)
                 | RCC_PLLCFGR_PLLSRC_HSE; // Источник PLL = HSE

    // ── Шаг 5: Включить PLL ──
    RCC->CR |= RCC_CR_PLLON;
    while (!(RCC->CR & RCC_CR_PLLRDY));   // Ждать готовности PLL

    // ── Шаг 6: Переключить SYSCLK на PLL ──
    RCC->CFGR |= RCC_CFGR_SW_PLL;         // SYSCLK = PLLCLK
    while ((RCC->CFGR & RCC_CFGR_SWS) != RCC_CFGR_SWS_PLL);
    // Ждать, пока переключение завершится (SWS = PLL)

    // ── Шаг 7 (опционально): Включить CSS ──
    // RCC->CR |= RCC_CR_CSSON;
}

int main(void) {
    SystemClock_Config();
    // Теперь SYSCLK = 168 МГц, HCLK = 168 МГц, APB1 = 42 МГц, APB2 = 84 МГц

    // Только ТЕПЕРЬ можно включать периферию
    RCC->AHB1ENR |= RCC_AHB1ENR_GPIOGEN;  // GPIOG
    GPIOG->MODER |= GPIO_MODER_MODE13_0;  // PG13 = Output

    while (1) {
        GPIOG->ODR ^= GPIO_ODR_OD13;
        for (volatile int i = 0; i < 1000000; i++);
    }
}
```

## 20. Типичные ошибки при настройке тактирования

| Ошибка | Симптом | Решение |
|--------|---------|---------|
| Не включил HSE перед PLL | PLL не блокируется, SYSCLK = HSI (16 МГц) | `while (!(RCC->CR & RCC_CR_HSERDY));` |
| Не настроил Flash latency | HardFault при любой записи/чтении Flash | `FLASH->ACR |= FLASH_ACR_LATENCY_5WS;` |
| PLLN за пределами 100-432 МГц | PLL не работает | Проверьте формулу: вход × PLLN = 100-432 МГц |
| SYSCLK > 168 МГц (STM32F4) | Нестабильная работа, зависания | Максимум 168 МГц для F4 |
| APB1 > 42 МГц | Периферия APB1 работает неверно | Максимум 42 МГц для APB1 |
| APB2 > 84 МГц | Периферия APB2 работает неверно | Максимум 84 МГц для APB2 |
| Забыл включить RCC_EN | Регистры периферии не отвечают | Сначала RCC_EN, потом настройка |
| Таймер работает на ×2 частоте | ШИМ на неверной частоте | Помните: TIM на APB1 = PCLK1 × 2 |
| PLLM даёт < 1 МГц или > 2 МГц | VCO не работает | PLLM должен дать 1-2 МГц на входе VCO |
